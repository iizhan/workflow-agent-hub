## 2026-05-21 网关选择态与多网关行为
### 这轮问题
- “已选择”容易被误解为“已启动”或“正在服务单聊”，需要更明确。
- 多个 AI 网关同时启动后，用户需要知道请求到底会走哪个网关。
- AI 网关右上角“重试/刷新”语义不清，容易被理解成重新发送消息或重启网关。

### 这轮结论
- “已选中”只代表右侧详情区当前正在查看/操作该网关，不代表它正在运行，也不代表它是单聊默认网关。
- 多个 AI 网关可以同时运行，彼此独立；单聊默认走当前激活 profile 对应的网关，群聊 Agent 走自己绑定 profile 对应的网关。
- 启动多个网关不会自动分流，也不会互相覆盖模型。
- 右上角操作应命名为“重新检查状态”，含义是重新拉取 Web UI、AI 网关、群聊绑定的最新状态，不做重启、不重发消息。

### 追加修正
- 网关管理的详情区不放“切换模型”，否则会混淆 `网关运行态管理` 和 `模型管理` 的职责。
- “已选中”保留在左侧列表即可，右侧详情重复展示会造成解释过度。
- 网关详情可以展示当前绑定模型，但不承担模型切换操作；模型增删改和切换统一进入 `模型管理` Tab。

## 2026-05-30 应用首页分流判断层收口
### 这轮需求
- 用户希望继续面向外部用户打磨当前项目，不只是补功能，而是持续检查产品流程和首页入口是否真的能带着用户往下走。
- 当前高曝光页面里，`Applications` 首页虽然已经有筛选和卡片，但仍然缺少“现在最值得先打开哪条应用、为什么”的第一眼判断层。

### 这轮结论
- `Applications` 首页不应只是应用目录，而应承担“应用分流台”的职责。
- 首页优先级判断顺序先按真实推进风险来排：
  - 等待审核
  - 最近失败
  - 正在正式执行
  - 已 ready 可启动
  - 仍需补起始配置
- 顶部需要显式给出一条“优先处理应用”主判断卡，直接告诉用户：
  - 现在先打开哪条应用
  - 为什么先打开它
  - 点下去后会进入哪个最合适的分区
- 列表卡片本身也要继续说真话：
  - 不是只显示状态
  - 而是显示这条应用当前属于“先处理审核 / 先排查失败 / 先关注现场 / 先准备开跑 / 先补齐起步”哪一类
- 点击应用卡不应总是统一回总览，而应尽量直接落到最合适的分区：
  - 审核、失败、ready 优先落到 `Runs`
  - live execution 优先落到 `Collaboration`
  - 已完成优先落到 `Artifacts`
  - 起步阶段再回 `Overview`

### 产品意义
- 这一步很关键，因为外部用户进入产品后首先面对的不是“功能是否存在”，而是“我现在该先开哪条应用”。
- 如果首页只能看见一堆卡片和状态，用户仍然要自己理解系统逻辑；只有把优先级和去向直接说出来，首页才真正成为工作台入口。

### 风险与下一步
- 当前这一轮先解决首页判断与卡片优先级露出，仍未处理 `Logs / Usage` 这类 legacy 页面里的产品化解释层。
- 下一步更适合继续把 legacy 页面从后台视角改成“为什么来这里看、看完后回哪处理”的辅助入口。
