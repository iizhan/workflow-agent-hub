# 领域模型校验

## 1. 本轮目的

在目标、成功标准和阶段边界已经明确之后，这一轮只做一件事：

把新产品的核心对象正式校验清楚，回答：

- 哪些是顶层对象
- 哪些是从属对象
- 哪些对象在当前系统里职责重叠
- 哪些对象要保留、演进、降级或逐步退出中心位置

本文件不进入页面结构和数据库实现，只做对象模型层的判断。

## 2. 领域模型校验标准

一个对象要被保留下来，至少要满足以下条件之一：

- 它是用户能理解的业务对象
- 它承载稳定的业务边界
- 它对跨场景复用有价值
- 它不能被其他对象自然吸收

如果一个对象只是历史实现残留、系统内部技术概念，或与另一个对象高度重叠，就不应继续作为顶层对象存在。

## 3. 顶层对象结论

经过当前阶段校验，建议新产品的稳定顶层对象为：

- `Application`
- `Project`
- `Agent`
- `Workflow`
- `Artifact`
- `Run`

以下对象应保留，但不应作为用户第一视角的顶层业务入口：

- `Skill`
- `Gateway`
- `Provider`
- `Model`
- `Profile`
- `Session`

以下对象应被重新定位：

- `Room`

## 4. 核心对象定义

### 4.1 Application

`Application` 是新的顶层业务容器，也是用户第一入口。

它负责组织：

- 目标与任务上下文
- 项目绑定
- 参与的 Agent
- 采用的 Workflow
- 产物集合
- 运行历史
- 对外展示的业务身份

### 为什么它必须是顶层

- 用户是围绕一个目标持续工作，而不是围绕系统配置工作
- 它能统一承载代码、文档、PPT、研究等多场景
- 它能吸收当前分散在群聊、项目绑定、工作流配置里的业务入口职责

### 不能把它降级为普通配置对象的原因

- 一旦 `Application` 不是顶层，用户又会重新面对大量分散对象
- 后续的页面、品牌、交付主线都会重新失焦

## 4.2 Project

`Project` 是真实资产源。

它代表：

- 代码仓库
- 本地目录
- 文档目录
- PPT 资料目录
- 研究资料目录
- 混合型资产目录

### 它的边界

- `Project` 负责“资产是什么、在哪里、能访问到什么”
- `Project` 不负责“谁来执行、按什么流程执行”
- `Project` 不负责“使用哪个模型”

### 和 Application 的关系

- `Application` 是工作容器
- `Project` 是被接入的真实资产
- 一个 `Application` 至少有一个主 `Project`
- 后续可扩展为一个主项目加多个附属项目

## 4.3 Agent

`Agent` 是角色执行单元。

它由以下信息组成：

- 角色名称
- 职责描述
- 提示词
- 技能边界
- 模型策略
- 权限范围

### 它的边界

- `Agent` 负责“谁来做”
- `Agent` 不负责“整个流程怎么走”
- `Agent` 不等于 `Model`
- `Agent` 不等于 `Gateway`

### 为什么要保留 Agent，而不是直接叫机器人或模型

- `Agent` 能自然承载角色语义
- 对内可持续表达执行能力、提示词、技能、模型策略组合
- 对外可以继续展示为“机器人”，但内部模型不要混乱

## 4.4 Workflow

`Workflow` 是应用级编排规则。

它负责：

- 流程节点
- 角色接力
- 审批与确认
- 产物要求
- 流转条件

### 它的边界

- `Workflow` 负责“怎么流转”
- 不负责“资产在哪里”
- 不负责“具体谁配置了什么模型”

### 和 Agent 的关系

- Workflow 决定由哪个 Agent 在哪个节点执行
- Agent 是执行者
- Workflow 是组织执行者的规则

## 4.5 Artifact

`Artifact` 是执行产物。

应统一承载：

- 代码变更建议
- 设计文档
- PRD
- 报告
- PPT 大纲
- 测试结果
- 表格和图片

### 为什么 Artifact 是顶层对象

- 它是用户最终能交付和回看的一等结果
- 它是跨场景最稳定的统一抽象
- 它能把对话型输出转换为项目型成果

## 4.6 Run

`Run` 是一次应用级任务执行。

### 它的价值

- 把“操作过程”从静态配置中分离出来
- 便于记录谁触发、何时执行、产生哪些产物
- 便于后续审计、回放、恢复和自动化

### 和 Session 的区别

- `Run` 是业务执行视角
- `Session` 是会话或过程上下文视角
- 一个 `Run` 可以关联多个 `Session`

## 5. 基础设施对象重新定位

## 5.1 Skill

`Skill` 不是顶层业务容器，而是能力供给层。

### 它应被如何理解

- 是被 `Application`、`Agent` 调用和装配的能力单元
- 可以存在作用域：
  - 全局
  - 应用级
  - Agent 级

### 它不能替代什么

- 不能替代业务流程
- 不能替代应用本身
- 不能替代项目上下文

## 5.2 Gateway

`Gateway` 是运行时执行基础设施。

### 它应被如何理解

- 负责承接 Hermes 执行链路
- 负责运行时能力联通
- 属于平台执行面

### 它不能替代什么

- 不能替代业务应用
- 不能替代 Agent 角色
- 不能替代项目边界

## 5.3 Provider / Model

### Provider

- 模型来源和凭证集合

### Model

- 具体可调用模型

### 关系判断

- Provider / Model 属于能力与资源层
- Agent 只绑定模型策略，不直接等于模型

## 5.4 Profile

`Profile` 更接近运行时配置隔离单元。

它应继续存在，但不应成为新产品的业务入口。

## 5.5 Session

`Session` 是过程上下文对象。

它应保留，但不再承担顶层业务容器职责。

## 6. Room 的重新定位

`Room` 是当前系统里最需要重新定位的对象。

### 当前问题

现在 `Room` 同时承担了：

- 协作空间
- 群聊入口
- 项目绑定容器
- 工作流容器
- Agent 容器

这导致它和未来的 `Application` 高度重叠。

### 校验结论

`Room` 不应继续作为长期顶层业务容器。

### 推荐定位

短期：

- `Room` 仍保留为现有运行载体
- 可视为 `Application` 的历史前身或内部协作视图

中期：

- `Application` 成为用户第一入口
- `Room` 降级为应用内的“协作模式 / 实时会话空间”

长期：

- 用户优先感知 `Application`
- `Room` 仅在需要实时多人协作、群聊式推进时显式出现

## 7. 对象关系结论

推荐的关系如下：

- `Application`
  - 绑定一个主 `Project`
  - 包含多个 `Agent`
  - 配置一个默认 `Workflow`
  - 生成多个 `Run`
  - 沉淀多个 `Artifact`

- `Run`
  - 关联一个 `Application`
  - 驱动一个具体任务执行过程
  - 关联多个 `Session`
  - 产出多个 `Artifact`

- `Agent`
  - 挂在 `Application` 下
  - 使用 `Skill`
  - 采用 `Model Strategy`
  - 通过 `Gateway` 执行

- `Workflow`
  - 挂在 `Application` 下
  - 调度多个 `Agent`
  - 定义节点、审批与产物要求

- `Project`
  - 被 `Application` 接入
  - 为 `Agent` 和 `Workflow` 提供上下文与落盘目标

## 8. 当前对象冲突处理建议

### 冲突 1：Application vs Room

结论：

- `Application` 取代 `Room` 的顶层业务容器地位
- `Room` 转为应用内协作空间语义

### 冲突 2：Agent vs Model

结论：

- `Agent` 是角色
- `Model` 是资源
- 二者严禁混用命名

### 冲突 3：Workflow vs Agent

结论：

- Workflow 组织执行
- Agent 负责执行

### 冲突 4：Project vs Artifact

结论：

- `Project` 是输入与落盘资产容器
- `Artifact` 是执行输出物
- 产物可以写回项目，但不等于项目本身

## 9. 演进策略建议

### 短期

- 保留现有 Room、Project、Workflow、Group Chat 能力
- 在产品与文档层引入 `Application` 作为第一概念

### 中期

- 新页面和导航以 `Application` 为主
- 现有房间能力被收编为应用内部协作模块

### 长期

- 数据模型逐步从 Room-first 过渡到 Application-first
- 用户不需要先理解历史概念就能开始工作

## 10. 本轮结论

本轮领域模型校验后的稳定判断是：

- `Application` 必须成为新的顶层业务容器
- `Project / Agent / Workflow / Artifact / Run` 构成主要业务主干
- `Skill / Gateway / Provider / Model / Profile / Session` 属于支撑层
- `Room` 需要从顶层业务容器降级为协作空间语义

只有先把这个对象模型稳定下来，后续信息架构、品牌重构和实现迁移才不会反复返工。
