# 信息架构设计

## 1. 本轮目的

在目标、成功标准、阶段边界和领域模型都已经明确之后，这一轮只解决一个问题：

新产品的信息架构应该如何组织，才能让 `Application` 成为用户第一入口，并把当前分散的功能页重新收拢为一条清晰主线。

本文件不进入界面视觉细节，也不进入前端路由实现，只定义页面层级、导航分区和主要入口关系。

## 2. 当前信息架构问题

当前系统更接近“系统功能导航”，而不是“用户任务导航”。

主要问题有：

- 用户进入后首先面对的是很多系统页，而不是工作入口
- 聊天、群聊、项目、工作流、模型、网关都在同一层竞争注意力
- 项目能力附着在群聊内部，难以成为通用工作台的一部分
- 群聊、会话、任务、产物之间缺少统一归属
- 系统级能力和业务级能力混在同一侧边栏中

## 3. 信息架构目标

新的信息架构必须满足以下要求：

### 3.1 入口清晰

用户第一次进入产品时，应先看到“应用”而不是“功能列表”。

### 3.2 主线明确

用户可以顺着一条稳定主线工作：

1. 找到或创建应用
2. 进入应用
3. 查看项目、角色、工作流和产物
4. 发起运行
5. 回看历史和结果

### 3.3 分层明确

业务对象和系统对象必须分层：

- 业务层：用户工作的地方
- 资源层：辅助工作所需资产与能力
- 系统层：基础设施和平台配置

### 3.4 可扩展

未来增加文档、PPT、研究、运营等场景时，不需要重做整套导航。

## 4. 顶层信息架构结论

建议新产品采用三层结构：

### 顶层一：应用层

围绕用户目标组织，是产品主入口。

### 顶层二：运行与资源层

围绕执行过程和通用资源组织，用于支撑应用内工作。

### 顶层三：系统层

围绕基础设施和平台配置组织，退到管理位置。

## 5. 推荐顶层导航

建议顶层导航分为四大域：

- `Applications`
- `Runs`
- `Resources`
- `System`

## 6. Applications 域

这是新的核心入口。

### 6.1 应用列表

作用：

- 展示所有应用
- 创建新应用
- 按场景筛选应用
- 展示应用状态、主项目、最近运行、最近产物

### 6.2 应用详情

应用详情页应成为最重要的业务页面。

它不是一个单页堆砌，而是一个“应用内导航容器”。

### 6.3 应用详情内部分区

建议至少包含以下分区：

- `Overview`
- `Projects`
- `Agents`
- `Workflow`
- `Artifacts`
- `Runs`
- `Collaboration`
- `Settings`

## 7. 应用详情分区定义

## 7.1 Overview

展示应用的整体状态。

应包含：

- 应用目标简介
- 主项目概览
- 参与 Agent 摘要
- 当前工作流摘要
- 最近运行
- 最近产物

这个页面承担“用户进入应用后的第一屏”职责。

## 7.2 Projects

这是 `Project` 的业务入口。

应包含：

- 主项目
- 附属项目
- 项目接入状态
- 目录浏览
- Git 状态摘要
- 项目权限概览

### 关键判断

项目不再作为群聊的附属弹窗存在，而应成为应用内一级分区。

## 7.3 Agents

这是 `Agent` 的业务入口。

应包含：

- 角色列表
- 每个角色的职责、技能、模型策略
- 角色启用状态
- 与工作流节点的关系

### 关键判断

模型页不再承担“用户配置业务角色”的主要职责，角色配置应收回到应用内。

## 7.4 Workflow

这是 `Workflow` 的业务入口。

应包含：

- 工作流概览
- 节点编排
- 角色分配
- 审批规则
- 产物要求

### 关键判断

工作流不再主要依附于群聊房间，而应作为应用内核心分区存在。

## 7.5 Artifacts

这是 `Artifact` 的业务入口。

应包含：

- 产物列表
- 按类型过滤
- 关联运行
- 关联项目写回位置
- 预览和版本摘要

### 关键判断

产物必须从“聊天中的附件或文本结果”升级为应用内一级信息资产。

## 7.6 Runs

这是 `Run` 的业务入口。

应包含：

- 运行列表
- 当前运行状态
- 运行日志
- 运行和产物的关系
- 运行和会话的关系

### 关键判断

运行是应用的过程视角，不再只散落在会话、Jobs 或群聊中。

## 7.7 Collaboration

这是 `Room` 重新定位后最合理的落点。

应包含：

- 群聊房间
- 协作成员
- 实时讨论
- 实时推进中的工作流消息

### 关键判断

群聊不再是产品主入口，而是应用内的协作视图。

## 7.8 Settings

这是应用级配置区域。

应包含：

- 应用元信息
- 默认技能策略
- 默认模型策略
- 权限策略
- 默认运行设置

## 8. Runs 域

`Runs` 域不是应用详情的替代，而是跨应用查看运行过程的统一入口。

### 适合放在这里的内容

- 全局运行列表
- 跨应用运行状态
- 失败运行
- 待审批运行
- 最近运行趋势

### 不适合放在这里的内容

- 应用内项目配置
- 应用内角色配置
- 应用内工作流建模

## 9. Resources 域

`Resources` 域承载跨应用复用的内容。

建议包含：

- `Skills`
- `Memory`
- `Files`
- `Templates`

### Skills

全局技能库与技能浏览入口。

### Memory

全局或用户级长期记忆入口。

### Files

跨应用文件资源入口，适合作为通用资源浏览层，而非主要项目入口。

### Templates

用于沉淀：

- 应用模板
- Agent 模板
- Workflow 模板
- 产物模板

## 10. System 域

`System` 域专门安放基础设施能力。

建议包含：

- `Gateways`
- `Models`
- `Providers`
- `Channels`
- `Profiles`
- `Logs`
- `Settings`

### 关键判断

这些页面仍然重要，但应明确退居管理层，而不是继续充当用户主工作入口。

## 11. 当前功能到新架构的映射建议

### Chat

当前定位：独立入口  
新定位：可作为应用内某类运行或会话视图的一部分，或作为 Runs / Collaboration 的子能力保留。

### History

当前定位：独立入口  
新定位：并入 `Runs` 域。

### Group Chat

当前定位：独立主入口  
新定位：并入应用详情下的 `Collaboration`。

### Jobs

当前定位：独立主入口  
新定位：短期保留在 `Runs` 域，长期可与运行调度能力合并。

### Kanban

当前定位：独立主入口  
新定位：可作为应用内或 Runs 域的工作管理视图，后续再决定是否作为应用模块挂载。

### Models / Channels / Gateways / Profiles

当前定位：主导航功能  
新定位：全部退到 `System` 域。

### Skills / Memory / Files

当前定位：主导航功能  
新定位：归入 `Resources` 域。

### Project 相关能力

当前定位：Group Chat 内部项目弹窗  
新定位：升级为 `Application -> Projects` 一级分区。

### Workflow 相关能力

当前定位：Group Chat / Room 内部配置  
新定位：升级为 `Application -> Workflow` 一级分区。

## 12. 首页策略建议

新首页不应直接跳到聊天或某个系统页。

建议首页默认进入：

- `Applications`

首页应优先呈现：

- 最近应用
- 推荐创建应用
- 最近运行
- 最近产物
- 模板入口

这样首页表达的是“开始工作”，不是“开始配置系统”。

## 13. 导航优先级结论

未来导航优先级应调整为：

1. 应用
2. 运行
3. 资源
4. 系统

而不是：

1. 聊天
2. 模型
3. 网关
4. 系统配置

## 14. 本轮结论

信息架构层最重要的结论是：

- `Application` 必须成为新的第一入口
- `Project / Agent / Workflow / Artifact / Run` 必须被收进应用内部主分区
- `Group Chat / Room` 应退化为应用内协作视图
- `Skills / Memory / Files` 应提升为资源层
- `Gateways / Models / Profiles / Channels` 应退居系统层

只有先完成这层收拢，后续界面风格重做和实现迁移才有稳定骨架。
