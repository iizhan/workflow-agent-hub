# 创建应用向导与首轮引导流程设计

## 1. 本轮目的

这一轮设计要解决：

- 用户如何从首页真正开始第一次创建应用
- 如何把当前系统已经存在的房间、项目、工作流、角色配置能力，重组成一条顺滑的创建主线
- 如何让 `Code / Document / PPT / Research` 不同场景在首轮引导中就有差异化起步方式

本轮不是凭空设计一个全新系统，而是基于当前已有实现做产品级重组。

## 2. 当前已有系统基础

经过现有代码检查，当前系统已经有以下可复用基础：

### 2.1 房间创建入口

现有 `CreateRoomForm.vue` 已具备：

- 用户名
- 用户描述
- 房间名
- 邀请码
- 基础 workflow 配置
- compression 配置

这说明当前系统已经有“创建协作容器”的入口，但它还是 `Room-first`。

### 2.2 房间内项目绑定

当前 `GroupChatPanel.vue` 已提供项目弹窗，支持：

- 绑定本地项目
- 绑定已有项目
- 查看项目文件
- 查看 Git 状态和分支

这说明 `Project` 能力已经可用，但位置不对，仍然附着在群聊内部。

### 2.3 房间内工作流配置

当前 `GroupChatPanel.vue` 已提供工作流配置弹窗，支持：

- 选择 workflow template
- workflow prompt
- stage-gated / freeform 模式
- owner role / owner user
- LogicFlow 画布配置
- 角色对齐检查

这说明 `Workflow` 能力已经相对成熟，但也是 `Room-first` 的。

### 2.4 房间内 Agent 管理

当前系统已支持：

- 添加 Agent
- 编辑 Agent
- 默认 Agent 保存与套用
- 角色与 workflow 对齐提示

这说明 `Agent` 已有可用的配置底座。

## 3. 当前痛点总结

当前的真实问题不是能力缺失，而是创建路径割裂：

- 用户先创建房间
- 再进群聊页面
- 再单独打开项目弹窗
- 再单独打开工作流弹窗
- 再单独添加 Agent

这使得首轮体验的问题是：

- 用户不知道先做哪一步
- 用户看不到完整应用的搭建进度
- 用户必须理解“房间”而不是“应用”
- 项目、流程、角色的关系只在脑中拼接

## 4. 设计结论

新产品的“创建应用”不应是一个简单表单，而应是一个有状态的向导流程。

这个向导的目标不是收集所有配置，而是帮助用户完成“第一次可运行的闭环搭建”。

## 5. 向导设计原则

### 原则 1：先建应用壳，再补运行闭环

不要在第一步就塞进全部复杂配置。

### 原则 2：优先复用现有模块心智

现有的：

- 房间创建
- 项目绑定
- Agent 配置
- Workflow 配置

都应尽可能演进复用，而不是完全改写概念。

### 原则 3：首轮只要求最小可运行闭环

第一次创建不需要配完所有高级能力，只要达到：

- 有应用名
- 有一个场景
- 有主项目
- 有至少一个 Agent
- 有一个工作流模板或默认流程

就可以让用户开始第一次运行。

### 原则 4：不同场景要有不同默认项

`Code / Document / PPT / Research` 的默认模板和引导不能完全相同。

## 6. 创建应用向导总流程

推荐采用 5 步向导：

1. 选择场景
2. 定义应用
3. 绑定项目
4. 配置角色
5. 选择流程并完成

## 7. 向导步骤设计

## 7.1 第一步：选择场景

### 页面目标

让用户快速进入最接近自己任务的起点。

### 推荐选项

- Code
- Document
- PPT
- Research

### 每个场景应默认注入什么

#### Code

- 默认推荐代码项目接入
- 默认推荐开发型 Agent 模板
- 默认推荐需求 -> 设计 -> 开发 -> QA 流程

#### Document

- 默认推荐文档目录接入
- 默认推荐作者 / 审校 / 审批角色
- 默认推荐写作与评审流程

#### PPT

- 默认推荐资料目录接入
- 默认推荐提纲 / 文案 / 审阅角色
- 默认推荐逐页产物流程

#### Research

- 默认推荐研究资料接入
- 默认推荐分析 / 核验 / 总结角色
- 默认推荐收集 -> 提炼 -> 结论流程

### 当前系统可复用点

- 现有尚无“场景选择”页面
- 但后续可以把它作为创建房间前的新入口层

## 7.2 第二步：定义应用

### 页面目标

让用户定义“这个应用是干什么的”。

### 推荐字段

- 应用名称
- 一句话目标
- 可选描述
- 可选标签

### 当前系统可复用点

- `CreateRoomForm.vue` 中的 `roomName`、`description` 可部分演进为应用名称与目标描述

### 设计变化

- 不再强调邀请码
- 不再先暴露房间概念
- 用户先看到的是“应用”，不是“群聊房间”

## 7.3 第三步：绑定项目

### 页面目标

帮助用户建立应用的真实资产上下文。

### 推荐能力

- 绑定本地项目
- 选择已有项目
- 查看项目连接是否成功

### 首轮必要信息

- 主项目路径或已有项目
- 项目类型
- 基础可访问性检查

### 当前系统可复用点

- 现有 `showProjectModal`
- 现有项目列表、项目详情、文件树、Git 状态能力
- `store.attachLocalProjectToRoom`
- `store.attachExistingProjectToRoom`

### 设计变化

- 项目绑定由群聊内部弹窗，前置为创建应用的第三步
- 让“主项目”成为应用创建闭环的一部分

## 7.4 第四步：配置角色

### 页面目标

帮助用户快速拥有第一批可执行角色，而不是从空白开始。

### 推荐能力

- 直接套用默认角色模板
- 再允许少量调整

### 推荐首轮最小要求

- 至少 1 个 Agent
- 最好 2-4 个典型角色，按场景给默认模板

### 当前系统可复用点

- 现有 Agent 配置能力
- 默认 Agent 保存与套用
- 角色对齐能力

### 设计变化

- 首次创建时优先走“模板 + 轻编辑”
- 不要求用户第一次就深度写 system prompt

## 7.5 第五步：选择流程并完成

### 页面目标

让应用进入“可开始第一次运行”的状态。

### 推荐能力

- 选择 workflow template
- 查看简要流程预览
- 确认主角色、关键阶段、产物方向

### 当前系统可复用点

- 现有 workflow template library
- LogicFlow workflow editor
- workflow role alignment 检查

### 设计变化

- 创建阶段优先“选模板 + 看摘要”
- 高级流程编辑延后到应用详情页内再做

## 8. 完成页设计

向导最后不应只是提示“创建成功”，而应进入：

- 应用创建成功
- 已完成的配置摘要
- 下一步推荐动作

### 推荐 CTA

- `Start First Run`
- `Open Application`
- `Review Workflow`

## 9. 向导与现有系统的映射策略

推荐采用“概念升级 + 能力复用”的策略。

### 当前对象到未来向导的映射

- `CreateRoomForm`
  -> `Create Application / 基础信息步骤`

- `Project Modal`
  -> `Create Application / 绑定项目步骤`

- `Add Agent Modal`
  -> `Create Application / 配置角色步骤`

- `Workflow Modal`
  -> `Create Application / 选择流程步骤`

### 核心策略

- 先不急着彻底删除这些现有入口
- 先在新应用主线里把它们重新编排
- 等应用入口稳定后，再逐步把旧入口降级或收起

## 10. 首轮引导规则

创建应用后，用户第一次进入应用详情时，仍然可能不完整。

因此应保留一个“首轮引导层”：

- 项目是否已绑定
- Agent 是否已就绪
- Workflow 是否已选定
- 是否已发起第一次运行
- 是否已有第一份产物

### 推荐表现

- Overview 中的 setup checklist
- 状态卡与下一步 CTA

## 11. 本轮结论

基于当前已有系统，新的创建应用流程不应推翻现有能力，而应把已有的：

- 房间创建
- 项目绑定
- Agent 配置
- Workflow 配置

重组成一条更符合产品目标的向导主线。

一句话总结：

> 先把当前分散在群聊内部的配置动作，前置并重组为“创建应用向导”，再让用户从应用进入真正的长期工作流。
