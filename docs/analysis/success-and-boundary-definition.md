# 成功标准、非目标与阶段边界定义

## 1. 本轮目的

这份文档用于把上一轮的产品目标继续压实，重点解决三个问题：

- 什么叫这次方向是成功的
- 什么明确不在当前阶段做
- 后续每个阶段做到什么程度才算进入下一步

本文件不负责页面方案和实现细节，只负责给后续设计设硬边界。

## 2. 成功标准总原则

成功标准必须满足三个条件：

- 可被团队复述
- 可被设计和实现直接对照
- 可用来否决跑偏方案

如果一个“成功标准”只能当愿景口号，不能指导取舍，就不算真正有效。

## 3. 一级成功标准

### 3.1 认知成功

第一次接触产品的用户，应该能在短时间内理解：

- 这不是旧的系统控制台
- 我应该先创建一个应用
- 应用是我围绕项目持续工作的入口

如果用户仍然只能把产品理解为“聊天 + 模型 + 网关配置页集合”，说明重构失败。

### 3.2 主线成功

用户完成一项工作的最短主线应变成：

1. 创建应用
2. 接入项目
3. 配置角色与工作流
4. 发起任务
5. 产生产物

如果主线仍然需要用户先理解很多系统对象，说明主线不清晰。

### 3.3 场景泛化成功

同一套顶层对象必须能同时解释至少四类场景：

- 程序开发
- 文档写作
- PPT 制作
- 研究分析

如果某个方案只能解释开发场景，说明抽象还不够通用。

### 3.4 沉淀成功

系统输出不能只停留在临时会话中，而应稳定沉淀为：

- 项目上下文
- 运行记录
- 产物
- 模板
- 可复用配置

如果用户仍然主要依赖复制聊天内容，说明产品没有形成真正的执行闭环。

### 3.5 品牌升级成功

新名称、新导航和新界面应共同让用户感知到：

- 这是一个通用项目执行工作台
- 不是旧的 Hermes 管理后台
- 不是单点 AI 工具

如果换了视觉但认知没变，说明品牌升级失败。

## 4. 二级成功标准

### 4.1 对产品设计的要求

- 每个顶层页面都能归属到清晰的业务层级
- 每个顶层对象都有明确边界
- 每个核心流程都能映射到应用主线

### 4.2 对交互设计的要求

- 首页优先展示“工作入口”而不是“系统配置入口”
- 应用详情页能承载项目、角色、工作流和产物
- 系统配置页退居到管理层而不是业务入口层

### 4.3 对后续实现的要求

- 能逐步复用现有群聊、项目、工作流基础能力
- 不要求一次性推翻全部历史实现
- 数据模型迁移必须支持演进，而不是断崖式替换

## 5. 当前阶段明确非目标

这一阶段明确不做以下事情：

### 5.1 不做完整重写

- 不重写全部前端页面
- 不重写全部后端服务
- 不直接废弃 Room、Group Chat、Project 现有能力

### 5.2 不做全量连接器扩张

- 不一次性支持所有远程 Git、Drive、Notion、Figma、飞书文档等连接器
- 不在目标和边界未稳定前就扩展太多接入源

### 5.3 不做高风险自动执行承诺

- 不承诺自动 commit / push / deploy 全链路
- 不在没有审批与审计模型前开放高风险远程写操作

### 5.4 不先做视觉表层翻新

- 不先为了“好看”做大面积样式替换
- 不在品牌定位未明确前先做完整 Logo 和视觉包装

### 5.5 不做伪通用化

- 不用很多“插件”“助手”“空间”之类空泛词汇掩盖对象边界不清的问题
- 不为了覆盖所有场景而提前引入过多抽象层

## 6. 阶段边界定义

## 6.1 阶段 0：目标与边界阶段

这一阶段要解决：

- 产品一句话目标
- 目标用户
- 核心问题
- 成功标准
- 非目标
- 顶层阶段边界

### 完成标准

- 团队能用一致语言描述产品目标
- 能明确说出当前不做什么
- 后续设计讨论有统一判断依据

### 未完成前不能进入

- 领域模型细化
- 页面结构细化
- 视觉系统细化
- 技术迁移细化

## 6.2 阶段 1：领域模型阶段

这一阶段要解决：

- Application / Project / Agent / Workflow / Skill / Gateway / Model / Artifact 的正式边界
- Room 与 Application 的关系
- 现有对象如何演进

### 完成标准

- 每个顶层对象职责明确
- 主要冲突边界被说明
- 不同场景都能被同一套模型解释

## 6.3 阶段 2：信息架构与页面阶段

这一阶段要解决：

- 顶层导航怎么变
- 应用列表与应用详情如何组织
- 系统管理能力如何退居二级
- 群聊、项目、工作流、产物如何在应用内重组

### 完成标准

- 用户主线可被页面结构承接
- 页面分层不再互相抢职责
- 新旧入口关系清楚

## 6.4 阶段 3：视觉与品牌阶段

这一阶段要解决：

- 新产品命名方向
- 品牌语义
- 视觉语言
- 首页与关键核心页面的风格基线

### 完成标准

- 视觉能够表达“工作台”而不是“后台”
- 品牌认知不再绑定旧产品
- 核心页面风格统一

## 6.5 阶段 4：迁移与实施阶段

这一阶段要解决：

- 数据模型迁移
- 路由迁移
- 页面重构优先级
- 能力平滑演进策略

### 完成标准

- 有清晰的 MVP 路线
- 有可落地的实现顺序
- 有明确的保留、替换、过渡策略

## 7. 当前阶段最重要的判断问题

在进入下一轮之前，必须回答清楚：

1. 当前成功标准是否足够硬，能用来否决错误方向？
2. 当前非目标是否足够明确，能挡住需求蔓延？
3. 当前阶段边界是否足够清楚，能避免我们又提前进入实现？

如果这三条里有任何一条回答不清，就不应进入更细的设计层。

## 8. 本轮结论

当前阶段应把重点放在：

- 固定产品目标
- 固定成功标准
- 固定非目标
- 固定阶段边界

只有这些固定下来之后，后续的领域模型、信息架构、品牌升级和界面风格重做才会真正有依据。
