# 新一代应用中心化 AI 工作台产品设计

## 1. 设计背景

当前仓库仍以 `Hermes Web UI` 作为工程名存在，但未来产品不再沿用该名称。

当前系统已经具备聊天、群聊、网关、模型、技能、文件、任务、看板、项目绑定等多项能力，但这些能力主要以工具页方式分散存在。

现状问题不是功能不足，而是主线不够清晰：

- 用户难以理解“从哪里开始”
- 项目、机器人、工作流、模型之间的关系分散在不同页面
- 群聊房间承担了过多“业务容器”职责
- 现有能力更像“控制台”，还不像“可落地的项目执行平台”

本次设计目标是把产品从“旧的 Hermes 管理面板”提升为“通用项目执行工作台”，让它可接入代码项目、文档项目、PPT 项目、研究项目、运营项目等不同场景。

本轮还要同步纳入两项产品级变化：

- 品牌与命名体系重做
- 界面视觉语言与交互风格重做

## 2. 核心产品目标

### 2.1 产品定位

新产品的目标形态应为：

> 一个以“应用”为中心、可接入任意项目资产、可编排机器人与工作流、可持续产生产物的 AI 项目执行平台。

### 2.2 要解决的核心问题

- 把分散的 AI 配置、项目上下文、技能和执行入口收拢成统一工作单元
- 让 AI 围绕真实项目持续工作，而不是停留在对话层
- 让不同类型项目都能复用同一套执行框架
- 让执行过程可追踪、可审批、可沉淀、可复用

## 3. 用户痛点

### 3.1 对个人用户

- 同一件事要在聊天、文件、模型、网关、群聊之间来回切换
- 每次开新任务都要重新解释项目背景
- 不清楚哪个模型、哪个机器人、哪个技能在起作用
- AI 结果很多，但真正进入项目目录和产物库的很少

### 3.2 对团队用户

- 无法把“人类协作”和“AI 协作”放进统一流程
- 没有稳定的项目边界、权限边界和审批边界
- 很难复用一个成熟的机器人角色或工作流模板
- 难以审计谁读了什么、写了什么、触发了什么模型

### 3.3 对平台自身

- 产品导航以“系统功能”组织，而不是以“用户目标”组织
- 群聊房间、项目绑定、工作流编排、技能管理之间边界重叠
- 后续如果继续加 Git、审批、远程项目、文档连接器，复杂度会继续堆积
- 现有品牌名称过强绑定 Hermes 运行时，不利于向通用工作台定位升级
- 当前界面更接近系统后台，缺少新的品牌识别和面向多场景工作的产品气质

## 4. 先定目标，再做设计

为了避免设计和实现脱节，后续完整研发流程应固定为：

1. 目标定义
2. 成功标准定义
3. 非目标与范围边界定义
4. 领域模型与信息架构设计
5. 交互与视觉设计
6. 数据模型与迁移设计
7. 分阶段实施计划

本文件属于第 4 步之前的总设计母文档，但后续所有方案都必须回扣到“目标是否成立”。

## 5. 目标产品模型

## 5.1 新的顶层对象

建议将产品的顶层对象统一为 `Application`，中文可称为“应用”或“应用工作台”。

`Application` 是用户真正操作的业务容器，负责组织：

- 项目资产
- 机器人角色
- 工作流
- 技能集
- 模型策略
- 运行记录
- 产物

### 5.2 领域对象定义

#### Application

表示一个面向具体目标的执行工作台。

典型例子：

- 前端重构应用
- 产品需求分析应用
- 投融资路演 PPT 应用
- 行业研究报告应用

#### Project

表示被应用接入的真实资产源。

支持的资产形态不应只限于代码仓库，还应包括：

- 本地目录
- 远程 Git 仓库
- 文档目录
- 演示文稿目录
- 研究资料目录
- 混合内容工作目录

#### Agent

表示一个可执行角色。

Agent 不等于模型，而是：

- 角色名称
- 角色提示词
- 默认技能集
- 默认模型策略
- 可访问项目范围
- 在工作流中的职责

对外可继续使用“机器人”这个说法，对内建议统一保留 `Agent`。

#### Workflow

表示任务如何在不同角色、节点、审批和产物之间流转。

Workflow 不属于单个 Agent，而属于 Application。

#### Skill

表示可复用能力包。

建议分为三层：

- 全局技能：平台级通用能力
- 应用技能：只对某个应用开放
- Agent 技能：只对某个角色开放

#### Gateway

表示运行时执行面。

职责：

- 承接 AI 调用
- 管理 profile / runtime
- 连接 Hermes 执行链路

Gateway 不承载业务语义，只负责执行。

#### Provider / Model

- Provider：模型来源与凭证集合
- Model：具体可调用模型

Agent 绑定的是模型策略，不是写死的单一模型。

#### Run / Session / Artifact

- Run：一次应用级任务执行
- Session：一次会话或执行上下文
- Artifact：执行产物，如代码、PRD、Markdown、PPT 大纲、测试报告、表格、图片

## 6. 推荐的对象关系

建议采用以下关系：

- 一个 `Application` 可以绑定一个主项目，后续可扩展多个附属项目
- 一个 `Application` 可以包含多个 `Agent`
- 一个 `Application` 配置一个默认 `Workflow`
- 一个 `Application` 可以配置专属技能集
- 一个 `Agent` 可以绑定默认模型策略
- 一个 `Run` 在某个 `Application` 内启动
- 一个 `Run` 会驱动多个 `Session`、生成多个 `Artifact`
- `Gateway` 作为执行基础设施被 `Application` 或 `Profile` 选择

## 7. 通用场景适配能力

### 6.1 程序开发场景

- 接入本地或远程代码仓库
- 读取代码、配置、日志、测试结果
- 走需求、设计、开发、测试、发布工作流
- 产出代码、文档、测试报告、提交建议

### 6.2 文档写作场景

- 接入资料目录或知识库
- 维护大纲、章节、版本和审校流
- 多角色执行，如作者、审校、事实核验、排版
- 产出 PRD、报告、方案、手册

### 6.3 PPT 制作场景

- 接入资料、品牌规范、图片素材目录
- 通过工作流完成提纲、逐页文案、视觉建议、演讲稿
- 产出 Markdown 大纲、页面结构说明、图片素材清单、PPT 文件

### 6.4 研究分析场景

- 接入网页资料、PDF、表格、访谈纪要
- 使用研究型 Agent 收集、对比、提炼信息
- 产出研究报告、结论摘要、数据表格、引用清单

### 6.5 运营执行场景

- 接入活动资料、内容计划表、素材目录
- 用工作流串联策划、内容、审校、分发
- 产出公众号文案、海报文案、周报、执行清单

## 8. 统一执行链路

统一的执行链路应为：

1. 用户创建应用
2. 应用绑定一个或多个项目资产
3. 用户选择或配置工作流
4. 用户配置参与的 Agent
5. 为 Agent 绑定技能和模型策略
6. 用户发起一个任务或运行
7. 工作流触发某个 Agent
8. Agent 通过 Gateway 调用模型
9. Agent 读取项目上下文、技能上下文和历史上下文
10. 产生产物并写入产物区或项目目录
11. 必要时进入审批或人工确认
12. 最终沉淀为可回看、可复用的交付结果

## 9. 产品信息架构建议

当前导航更偏系统后台，建议逐步重构为以下结构：

### 顶层导航

- 应用
- 运行
- 资源
- 系统

### 应用域

- 应用列表
- 应用详情
- 项目
- Agent
- 工作流
- 产物

### 运行域

- 当前运行
- 历史会话
- Jobs
- Kanban

### 资源域

- Skills
- Memory
- Files
- Templates

### 系统域

- Gateways
- Models
- Channels
- Profiles
- Logs
- Settings

## 10. 品牌与视觉重构方向

这次不只是产品结构重构，还应同步做品牌与界面风格升级。

### 10.1 品牌方向

- 产品名不再沿用 `Hermes Web UI`
- 未来品牌名应弱化底层运行时实现，强化“应用工作台 / AI 执行平台 / 项目操作系统”认知
- 命名应适配多场景，而不是仅让人联想到聊天面板或某个单一 Agent 生态

### 10.2 界面风格方向

- 从“系统后台工具感”升级为“面向创作与执行的工作台感”
- 首页不再以功能菜单堆叠为中心，而应以应用入口、运行状态和产物流为中心
- 视觉上要建立新的品牌识别：
  - 更清晰的品牌主色和语义色
  - 更有辨识度的字体层级
  - 更强的卡片、画布、流程与产物视觉结构
  - 更适合多场景工作的桌面级布局

### 10.3 重构边界

- 本轮先定义品牌方向与视觉原则
- 新名字、Logo、完整视觉系统可以在后续单独拆成品牌与 UI 设计任务
- 但信息架构与页面设计现在就要按新品牌和新风格的目标来思考

## 11. 当前代码基可承接的演进方向

现有项目已经具备以下可复用基础：

- 群聊房间与 Agent 机制
- 工作流模板与流程节点设计
- 项目绑定、文件浏览、Git 状态读取
- Skills 浏览与 Memory 管理
- Gateway / Model / Profile 管理

因此不建议推翻重写，而建议采用“领域收拢 + 页面重构 + 数据模型升级”的方式迭代。

### 9.1 推荐演进原则

- 不直接推翻现有群聊能力
- 短期把 `Room` 视为 `Application` 的前身
- 中期把群聊从“顶层业务容器”降级为“应用内协作模式”
- 先重构概念与页面组织，再逐步升级底层数据模型

## 12. 分阶段路线图

### Phase 0：目标、命名与导航重构

目标：先把目标、命名方向、产品主线和导航讲清楚。

重点：

- 明确新产品不再沿用 `Hermes Web UI` 命名
- 明确 `Application / Project / Agent / Workflow / Skill / Gateway / Model` 边界
- 设计新导航与页面组织
- 定义新的界面风格方向
- 让“应用”成为用户第一入口

### Phase 1：Application MVP

目标：让用户能真正“创建一个应用并跑起来”。

范围：

- 创建应用
- 绑定主项目
- 配置参与 Agent
- 选择工作流模板
- 查看项目、会话和产物

### Phase 2：Workflow + Artifact 深化

目标：让应用具备真实执行闭环。

范围：

- 可视化工作流编辑
- 节点级角色、审批、模型覆盖
- 产物目录映射到项目
- 运行态可追踪

### Phase 3：通用连接器

目标：支持跨场景接入。

范围：

- 远程 Git 项目
- 云文档 / 本地文档目录
- PPT 目录 / 模板目录
- 网页研究输入源

### Phase 4：审计与自动化

目标：支持更高风险、更团队化的执行。

范围：

- 提交 / 推送审批
- 操作日志
- 运行回放
- 自动任务与守护式执行

## 13. 当前最优先要解决的产品问题

按优先级排序，当前最优先的问题是：

1. 顶层产品主线不清晰
2. 群聊、项目、工作流边界重叠
3. 项目能力还停留在群聊弹窗级
4. 缺少“应用”这一层统一业务容器
5. 不同场景的项目接入还没有统一抽象

## 14. 设计结论

这个产品的下一阶段不应该继续横向堆功能，而应该纵向收拢成一套“应用中心化”的执行模型，并同步完成品牌和界面风格升级。

一句话总结：

> 先把产品从“旧控制台”重构为“新的应用工作台”，同时建立新品牌和新界面语言，再让代码、文档、PPT、研究、运营等不同场景都通过统一的项目、Agent、工作流、技能、产物模型接入。
