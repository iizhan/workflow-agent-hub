# 现有系统到应用中心化产品的迁移策略设计

## 1. 本轮目的

前面的目标、边界、领域模型、信息架构、页面结构和状态规则已经基本明确。

这一轮不再讨论“理想中的新产品是什么”，而是只回答一个更现实的问题：

> 在当前已有系统基础上，怎样把旧的 `Hermes Web UI` 渐进迁移成新的应用中心化通用工作台。

本文件重点解决：

- 当前系统哪些能力已经可以直接复用
- 哪些地方只是命名和边界不对
- `Room` 如何短期继续存在、长期如何降级
- 第一阶段实现应该先改什么、不该先改什么
- 路由、前端页面、服务接口和数据模型应如何渐进演进

## 2. 当前系统现实基础

结合现有代码，当前系统已经具备一条可复用的业务主链路：

- 前端已有房间创建与进入能力
- 前端已有群聊协作面板
- 前端已有项目绑定、文件浏览、Git 状态查看
- 前端已有 Agent 增删改配置
- 前端已有工作流模板、工作流编辑与运行状态展示
- 前端已有产物浏览与审批流交互
- 服务端已有项目服务、房间服务、工作流运行状态、产物记录和项目绑定接口
- 数据库中已有 `gc_rooms / gc_room_agents / gc_room_projects / gc_workflow_runs / gc_workflow_artifacts / gc_projects` 等核心表

这说明当前系统的问题不是“没有能力”，而是：

- 顶层业务容器叫错了
- 页面入口放错了
- 业务对象边界混在一起
- 新用户首先看到的是系统功能，不是工作主线

因此迁移策略应采用：

- 渐进迁移
- 语义升维
- 页面重组优先
- 数据兼容优先

而不是：

- 一次性推翻重做
- 先做数据库大重构
- 先替换所有后端接口

## 3. 现有模块与未来模型的映射关系

## 3.1 Room 到 Application

短期判断：

- 当前 `Room` 已经承载了名称、描述、工作流、Agent、项目绑定、消息协作和运行上下文
- 它在能力上已经非常接近未来的 `Application`

问题在于：

- `Room` 的命名会让用户理解成“聊天房间”
- 它天然把产品重心拉向群聊，而不是工作台
- 它无法自然承载代码、文档、PPT、研究等多场景业务心智

所以建议：

- 第一阶段：在产品层引入 `Application` 概念，但底层继续映射到现有 `Room`
- 第二阶段：让 `Room` 退化为 `Application` 内的 `Collaboration` 子对象
- 第三阶段：视数据与代码稳定度，再决定是否新增独立的 `applications` 持久层

换句话说：

- 短期 `Application` 是对 `Room` 的业务包装
- 中期 `Application` 是业务主对象，`Room` 是协作运行时
- 长期 `Room` 只保留协作语义，不再代表顶层业务容器

## 3.2 现有页面到未来页面的映射

### 当前 `GroupChatView`

未来建议定位为：

- 应用详情中的 `Collaboration`
- 或者第一阶段的“应用工作台总页”

原因：

- 它已经有最完整的项目、Agent、Workflow、产物和运行态信息
- 它是当前最接近未来应用详情页的现成复用体

### 当前 `CreateRoomForm`

未来建议定位为：

- “创建应用”向导的底层能力来源

可直接复用的字段：

- 名称
- 描述
- 工作流基础配置
- 压缩配置

需要补的不是底层能力，而是上层步骤语义：

- 场景选择
- 项目绑定
- Agent 配置
- 模板选择

### 当前项目弹窗能力

未来建议定位为：

- 应用详情中的 `Projects`

当前弹窗里的本地项目绑定、已有项目绑定、文件浏览、Git 状态查看，已经是一个 `Projects` 分区的最小可用骨架。

### 当前工作流弹窗能力

未来建议定位为：

- 应用详情中的 `Workflow`

当前的模板管理、LogicFlow 编辑器、运行设置、审批状态展示都可以沿用。

### 当前 Agent 配置能力

未来建议定位为：

- 应用详情中的 `Agents`

当前已经具备角色名称、描述、头像、提示词、模型、温度等配置，不需要重做概念，只需要调整组织方式与命名。

### 当前产物与运行态能力

未来建议定位为：

- 应用详情中的 `Artifacts`
- 应用详情中的 `Runs`
- 以及全局 `Runs`

## 3.3 现有服务接口到未来边界的映射

### 可直接保留的接口族

- `/api/hermes/projects/*`
- `/api/hermes/group-chat/rooms/:roomId/project`
- `/api/hermes/group-chat/rooms/:roomId/agents*`
- `/api/hermes/group-chat/rooms/:roomId/workflow*`
- `/api/hermes/group-chat/rooms/:roomId/artifacts*`
- `/api/hermes/group-chat/rooms/:roomId/runtime`

这些接口的主要问题不是能力缺失，而是：

- URL 仍然暴露 `group-chat / rooms` 旧语义
- 不适合作为长期对外产品 API 语言

因此建议：

- 第一阶段：前端应用层先做包装，不急着替换底层接口
- 第二阶段：增加 `applications` 风格的新接口别名
- 第三阶段：旧接口进入兼容层，仅供旧页面使用

## 4. 复用、包装、替换矩阵

| 模块 | 当前状态 | 第一阶段策略 | 中长期策略 |
| --- | --- | --- | --- |
| 房间创建 | 已可用 | 包装为应用创建起点 | 最终替换为应用创建向导 |
| 群聊面板 | 功能最完整 | 重组为应用工作台主壳 | 退化为应用内协作分区 |
| 项目服务 | 已独立 | 直接复用 | 扩展多项目与远程连接器 |
| Agent CRUD | 已可用 | 直接复用并改文案语义 | 补权限、技能作用域、模板 |
| 工作流编辑器 | 已可用 | 直接复用 | 升级为应用级流程中心 |
| 工作流运行状态 | 已可用 | 直接复用 | 抽象为通用 Run 体系 |
| 产物浏览 | 已可用 | 直接复用并升为一级分区 | 支持多类型产物与版本视图 |
| 顶层导航 | 旧功能型 | 新建应用型入口 | 逐步隐藏旧导航 |
| Room 持久层 | 现有核心容器 | 暂不替换 | 后续拆分为 Application 与 Collaboration |

## 5. 第一阶段实施范围建议

第一阶段不应该一口气做“全平台重构”，而应先完成一个真实可见的产品迁移入口。

建议第一阶段只做以下范围：

### 5.1 新首页与新导航壳

新增应用中心化入口：

- `Applications`
- `Runs`
- `Resources`
- `System`

但第一阶段只要求真正做实：

- 应用列表
- 应用详情壳
- 应用创建入口

### 5.2 应用列表页

第一阶段可直接基于现有 `Room` 列表映射出应用卡片：

- 应用名 = 房间名
- 最近活动 = 房间最近运行或消息时间
- 主项目 = 当前绑定项目
- 角色数 = 房间 Agent 数
- 状态 = 通过房间工作流运行态推导

这样可以在不改数据库的前提下，先把“应用是第一入口”的产品感建立起来。

### 5.3 应用详情第一版

第一版应用详情页不要求一开始就拆成很多独立大页，可以先做一个新的应用壳页面，并把现有群聊面板能力按分区收进去：

- `Overview`
- `Projects`
- `Agents`
- `Workflow`
- `Artifacts`
- `Runs`
- `Collaboration`

第一阶段里，最务实的方式是：

- 先把 `GroupChatPanel` 作为底层复用容器
- 再逐步把它内部的大弹窗拆成应用内分区

### 5.4 创建应用向导第一版

第一版建议不要完整重写。

可以基于现有 `CreateRoomForm` 做如下包装：

1. 选择场景
2. 创建基础信息
3. 绑定项目
4. 配置角色
5. 选择工作流

其中只有第 1 步是新增表达层，后面几步大都可以复用现有能力。

## 6. 不建议第一阶段优先做的事情

为了避免研发节奏再次失焦，以下事情不建议作为第一阶段优先项：

- 先做数据库大迁移
- 先改所有服务端接口命名
- 先做完整多项目编排
- 先做远程 Git 全链路接入
- 先做复杂权限系统
- 先做高风险自动写入 / 自动提交 / 自动推送
- 先重写全部群聊逻辑

这些事情都重要，但它们应该建立在“应用入口已经成立”之后。

## 7. 路由迁移策略

当前路由仍是典型旧功能型结构，例如：

- `/hermes/chat`
- `/hermes/group-chat`
- `/hermes/models`
- `/hermes/gateways`

建议迁移为两层：

### 第一阶段新增，不立即删除旧路由

- `/workbench/applications`
- `/workbench/applications/:applicationId`
- `/workbench/runs`
- `/workbench/resources`
- `/workbench/system`

### 旧路由处理策略

- `/hermes/group-chat` 暂时保留，但从主导航降级
- `/hermes/models`、`/hermes/gateways` 等系统页继续保留，但归入 `System`
- `/hermes/chat`、`/hermes/history` 等旧页在第一阶段不主动删除

这样做的好处是：

- 新产品入口可以尽快建立
- 旧用户不被硬切断
- 团队可以先完成主线切换，再逐步清理历史页面

## 8. 数据模型迁移策略

## 8.1 第一阶段

不强推数据库重构。

采用“应用视图映射到房间数据”的方式：

- `Application.id = Room.id`
- `Application.name = Room.name`
- `Application.workflow = gc_rooms.workflow* + gc_workflow_runs`
- `Application.agents = gc_room_agents`
- `Application.projects = gc_room_projects + gc_projects`
- `Application.artifacts = gc_workflow_artifacts`

第一阶段只需要新增：

- 应用层视图模型
- 状态推导规则
- 前端聚合接口或前端聚合 store

## 8.2 第二阶段

当应用入口与页面结构稳定后，再考虑引入正式 `applications` 表，承接：

- 场景类型
- 目标描述
- 品牌化封面信息
- 默认视图配置
- 生命周期状态

此时 `Room` 可以改为：

- `applicationId`
- 协作房间元数据
- 实时协作状态

## 8.3 第三阶段

再决定是否把以下对象彻底抽出成独立通用模型：

- `runs`
- `artifacts`
- `application_agents`
- `application_projects`
- `application_workflows`

判断标准不是“结构是否优雅”，而是：

- 当前业务是否已经跨越 `Room` 语义极限
- 新旧数据兼容成本是否可控
- 前后端聚合层是否已经变得过重

## 9. 风险与控制

## 9.1 风险一：概念变了，底层没变，导致前后表达混乱

控制方式：

- 前端新增 `Application` 视图模型
- 旧词 `Room` 仅保留在内部兼容层和少量技术文档中
- 新页面、新文案、新导航统一使用 `Application`

## 9.2 风险二：群聊面板过重，应用详情页继续失控

控制方式：

- 第一阶段允许复用 `GroupChatPanel`
- 但必须把它视为临时过渡壳
- 后续要按 `Overview / Projects / Agents / Workflow / Artifacts / Collaboration` 逐步拆解

## 9.3 风险三：用户无法理解新旧入口关系

控制方式：

- 新首页只主推 `Applications`
- 旧入口降级或加“旧版/兼容入口”语义
- 在应用详情页中明确展示它与协作空间、项目、流程的关系

## 9.4 风险四：过早动数据库，影响现有稳定能力

控制方式：

- 第一阶段不做破坏性迁移
- 先做视图层和路由层重构
- 等应用主线验证稳定后再拆表

## 10. 第一阶段落地顺序建议

建议按以下顺序推进：

1. 定义新的产品命名占位与 UI 基线
2. 新建应用中心化导航壳
3. 做应用列表页
4. 做应用详情过渡壳
5. 把房间创建包装为应用创建向导第一版
6. 把项目 / Agent / Workflow 从弹窗能力逐步上收为应用分区
7. 最后再处理旧入口降级与兼容

## 11. 本轮最终结论

当前系统完全值得继续演进，不需要推翻。

真正要迁移的核心不是“已有能力”，而是以下四件事：

- 顶层容器从 `Room` 升维到 `Application`
- 顶层导航从“功能控制台”切换为“工作入口”
- 页面结构从“弹窗附属能力”升级为“应用内一级分区”
- 数据演进从“直接重构”改为“先映射、再拆分、后抽象”

这意味着后续实现应优先做：

- 新入口
- 新页面壳
- 新语义
- 现有能力重组

而不是优先做：

- 大规模底层重写
- 彻底替换现有房间体系
- 一步到位的数据结构重构
