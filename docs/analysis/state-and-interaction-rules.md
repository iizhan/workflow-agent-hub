# 关键状态与交互规则设计

## 1. 本轮目的

这份文档用于定义新产品在关键状态下的行为规则，确保后续页面实现不是只做“正常态”，而是从一开始就具备完整的体验骨架。

本轮重点回答：

- 用户第一次进入时看到什么
- 应用未完成配置时如何被引导
- 运行、审批、失败、空数据时如何反馈
- 用户如何从一个状态被引导到下一个正确动作

本文件不进入组件细节和实现逻辑，只定义状态系统与交互原则。

## 2. 状态设计总原则

### 原则 1：每个状态都要给出下一步

不能只告诉用户“现在还没有”，还必须告诉用户“接下来该做什么”。

### 原则 2：状态优先于功能展示

页面先表达当前状态，再暴露可做操作，而不是先摆一堆按钮。

### 原则 3：系统要帮助用户完成主线推进

产品的任务不是展示复杂性，而是帮助用户顺着：

创建应用 -> 接入项目 -> 配置角色 -> 配置流程 -> 发起运行 -> 产生产物

这条主线自然前进。

### 原则 4：反馈必须分级

不同状态反馈强度不同：

- 信息提示
- 引导提示
- 警告提示
- 阻断提示

不能全部用同一种 Toast 或同一种红字处理。

## 3. 全局关键状态模型

建议在应用级统一维护以下关键状态分类：

- `empty`
- `draft`
- `setup_required`
- `ready`
- `running`
- `waiting_review`
- `blocked`
- `failed`
- `completed`

## 4. 首页状态规则

## 4.1 首次进入产品

### 条件

- 用户没有任何应用

### 页面目标

- 让用户理解这是什么产品
- 帮用户开始第一次创建

### 应显示内容

- 一句话产品定位
- “创建应用”主按钮
- 场景入口：
  - Code
  - Document
  - PPT
  - Research
- 模板入口

### 不应显示内容

- 空白数据表
- 系统配置面板
- 复杂监控卡片

### 主 CTA

- `Create Application`

## 4.2 已有应用但无最近活动

### 条件

- 有应用
- 最近无运行或产物

### 页面目标

- 帮用户回到应用目录并推动第一次运行

### 推荐提示

- “你已有应用，但还没有开始第一次运行”
- 推荐进入最近创建的应用

## 4.3 有最近活动

### 条件

- 有应用
- 有最近运行或产物

### 页面目标

- 继续工作而不是重新开始

### 应优先展示

- 最近应用
- 最近运行
- 待处理事项
- 最近产物

## 5. 应用生命周期状态

## 5.1 Draft

### 含义

应用已创建，但还没有形成可执行闭环。

### 典型触发条件

- 未绑定主项目
- 未配置 Agent
- 未配置 Workflow

### 页面表现

- 应用头部显示 `Draft`
- Overview 顶部显示 setup checklist

### 主交互

- 引导用户按顺序完成配置

## 5.2 Setup Required

### 含义

应用存在关键缺口，无法正常开始运行。

### 必须阻断的条件

- 没有主项目
- 没有启用 Agent
- 没有可用 Workflow

### 交互规则

- “Start Run” 不能正常开始
- 点击后应解释缺少什么，并提供跳转入口

## 5.3 Ready

### 含义

应用具备基础执行条件。

### 页面表现

- 应用头部显示 `Ready`
- 主按钮可发起运行

### 推荐动作

- 启动第一次运行
- 或继续最近一次未完成工作

## 5.4 Running

### 含义

应用当前存在活跃运行。

### 页面表现

- 头部显示运行中状态
- Overview 与 Runs 模块高亮当前运行

### 交互规则

- 主 CTA 从“开始运行”切换为“查看当前运行”
- 不建议继续强推新的运行，除非用户明确允许并行

## 5.5 Waiting Review

### 含义

当前应用存在待审批、待确认或待处理产物。

### 页面表现

- 头部显示柔性警示
- Overview 显示待处理卡片
- Artifacts 或 Runs 中显示明确数量

### 主交互

- 一键跳转到对应待处理项

## 5.6 Blocked

### 含义

运行未失败，但因为某个外部条件无法继续。

### 常见原因

- 缺少角色
- 缺少权限
- 缺少审批
- 项目不可访问

### 页面表现

- 显示阻塞原因
- 给出恢复动作

## 5.7 Failed

### 含义

某次运行明确失败，无法自动继续。

### 页面表现

- 明确标记失败
- 展示失败位置和失败类型
- 不要只显示“请求失败”

### 主交互

- 查看错误详情
- 返回对应模块修复
- 重新运行

## 5.8 Completed

### 含义

当前运行链路已完成，且已有可交付结果。

### 页面表现

- 展示完成摘要
- 展示产物入口
- 展示下一次可复用入口

## 6. 应用内设置引导规则

## 6.1 未绑定主项目

### 出现位置

- Overview
- Projects
- Start Run 按钮交互

### 文案目标

- 不只提示缺失
- 要解释项目是应用上下文基础

### 推荐 CTA

- `Attach Primary Project`

## 6.2 未配置 Agent

### 出现位置

- Overview
- Agents
- Workflow 详情

### 推荐 CTA

- `Create First Agent`

### 特别规则

- 如果 Workflow 已存在但缺少角色匹配，应给出“角色未对齐”提示，而不是泛泛说配置不完整

## 6.3 未配置 Workflow

### 出现位置

- Overview
- Workflow
- Start Run 按钮交互

### 推荐 CTA

- `Choose Workflow Template`

## 6.4 未产生任何产物

### 出现位置

- Artifacts

### 页面目标

- 引导用户理解产物会在什么情况下出现

### 推荐提示

- “产物将在运行产生输出后出现在这里”
- 引导去发起运行或查看当前流程

## 7. 运行状态规则

## 7.1 首次运行前

### 页面目标

- 降低第一次发起运行的心理成本

### 推荐行为

- 给出预览说明：
  - 将使用哪个工作流
  - 哪些 Agent 会参与
  - 结果会去哪

## 7.2 运行中

### 页面目标

- 让用户知道系统正在做什么

### 应显示内容

- 当前运行名
- 当前阶段
- 当前参与角色
- 最近更新

### 不应只显示

- 一个纯 loading spinner

## 7.3 运行完成但待确认

### 页面目标

- 让用户迅速进入 review 模式

### 推荐 CTA

- `Review Artifact`
- `Approve Next Step`

## 7.4 运行失败

### 页面目标

- 从失败状态快速恢复

### 推荐动作

- 查看失败原因
- 跳转到配置缺口
- 重新运行

## 8. 审批与确认状态规则

## 8.1 待审批

### 含义

流程到达人工决策节点。

### 必须展示

- 为什么需要审批
- 谁应该审批
- 审批通过后会发生什么
- 驳回后会回到哪里

### 推荐 CTA

- `Approve`
- `Request Changes`

## 8.2 待确认产物

### 含义

产物已生成，但需要人确认是否可进入下一步或写回项目。

### 推荐展示

- 产物摘要
- 来源运行
- 目标落盘位置
- 当前确认状态

## 9. 空状态规则

## 9.1 Projects 空状态

- 应解释为什么要先绑定项目
- CTA：`Attach Primary Project`

## 9.2 Agents 空状态

- 应解释角色如何参与工作流
- CTA：`Create First Agent`

## 9.3 Workflow 空状态

- 应解释工作流控制什么
- CTA：`Use Template`

## 9.4 Artifacts 空状态

- 应解释产物何时出现
- CTA：`Start First Run`

## 9.5 Collaboration 空状态

- 应解释这里是实时协作视图
- CTA：`Open Collaboration`
  或引导先启动相关运行

## 10. 失败与阻断反馈分级

建议统一分为四级：

### L1 信息

- 用于普通说明
- 不打断主流程

### L2 引导

- 用于提示用户下一步
- 可附带 CTA

### L3 警告

- 用于待审批、配置缺口、风险提示
- 需要用户注意，但不一定立即阻断

### L4 阻断

- 用于无法运行、权限不足、关键资源缺失、运行失败
- 必须给出修复入口

## 11. 关键跳转规则

状态设计的核心不是提示本身，而是“提示后跳去哪”。

推荐固定以下跳转关系：

- 未绑定项目 -> `Projects`
- 未配置 Agent -> `Agents`
- 未配置 Workflow -> `Workflow`
- 待确认产物 -> `Artifacts`
- 运行中 -> `Runs`
- 协作阻塞 -> `Collaboration`

## 12. 本轮结论

状态系统层最重要的结论是：

- 产品不能只设计正常态
- 应用必须具备明确的生命周期状态
- 每个关键状态都要有下一步动作
- 阻断、审批、失败和空状态是工作台体验的核心组成，而不是补充细节

后续如果开始页面实现，必须优先把这些状态落进核心页面，而不是只做静态主界面。
