# 协作域高曝光文案最终收尾设计

## 1. 本轮目的

在协作与历史入口已经完成一轮语言对齐之后，本轮继续检查协作运行时中剩余的高曝光硬编码文案，确认哪些值得继续收口，哪些应保留为领域内准确术语。

## 2. 当前判断

继续扫描后发现，前台高曝光入口里最明显的问题已经基本清掉，剩余内容主要分两类：

- 组件内少量状态提示与按钮文案的硬编码
- 协作域内部术语，例如 `群聊 Agent`、`AI 网关`

第二类虽然仍带有旧能力来源，但在当前阶段仍然具有较强的语义准确性，不适合为了“去旧词”而强行改成更模糊的词。

## 3. 本轮设计结论

### 3.1 继续清理高曝光硬编码，但不强改准确领域术语

本轮确认的原则是：

- 继续清理高曝光硬编码状态词
- 不强行替换已经承担准确说明作用的协作域术语

### 3.2 “群聊运行时”与“协作工作区”可以分层共存

前台入口继续使用：

- 协作
- 协作工作区
- 运行时历史

而在协作深层功能中，仍允许保留：

- 群聊 Agent
- AI 网关

这样能同时满足产品语言升级和当前功能说明准确性。

## 4. 本轮结果

- 协作运行时里真正高曝光、应配置化的按钮与状态文案已经基本完成收口
- 剩余词汇主要是可解释的领域术语，而不是品牌层残留
- 前台后续可以把重点转向其他页面或下一批功能体验，而不是继续机械扫词

## 5. 下一阶段建议

1. 若继续扫词，优先只处理新的高曝光入口
2. 对协作域深层术语，后续应结合信息架构一起调整，而不是孤立替词
3. 把更多精力转向应用工作台未完成分区或交互细节
