# 下一阶段功能模块设计

## 1. 本轮目的

这一轮不再停留在“系统应该应用中心化”这种总方向，而是直接回答下一阶段到底该设计哪些功能模块。

核心目标：

- 把应用工作台拆成清晰模块
- 为每个模块定义要解决的问题
- 说明每个模块对用户的价值
- 明确首版做到什么程度
- 明确优先复用当前系统哪些能力

## 2. 设计原则

下一阶段的模块设计必须遵守以下原则：

- 先围绕真实工作过程设计，不围绕旧页面名称设计
- 先做应用主线，不先做系统配置搬家
- 先把用户最常进入的主页面做清楚，再拆深层能力
- 先复用现有 `Room / GroupChat / Project / Workflow` 内核，再逐步正式化
- 首版强调“能持续工作”，不是“所有分区都一次做满”

## 3. 下一阶段模块总览

建议下一阶段按八个模块推进：

- `Applications Home`
- `Application Overview`
- `Projects Workspace`
- `Agents Workspace`
- `Workflow Workspace`
- `Artifacts Workspace`
- `Runs Workspace`
- `Collaboration Workspace`

其中：

- 前两个模块负责建立应用主入口
- 中间四个模块负责把业务骨架拆出来
- 后两个模块负责承接执行过程与协作过程

## 4. Applications Home

### 4.1 要解决的问题

当前用户进入系统后，还不够自然地知道：

- 应该先进哪个应用
- 当前有哪些应用在推进
- 哪些应用卡住了
- 上次工作做到哪了

### 4.2 对用户的价值

`Applications Home` 的价值不是列卡片，而是帮助用户快速决定“接下来继续哪个应用”。

它应承接三类动作：

- 继续已有工作
- 创建新应用
- 发现需要关注的阻塞或待处理应用

### 4.3 首版范围

首版建议包含：

- 应用列表
- 搜索
- 场景筛选
- 状态筛选
- 创建应用入口
- 最近更新状态
- 基础空状态与下一步引导

### 4.4 优先复用的现有基础

- 现有 `ApplicationsView.vue`
- `ApplicationSummary` 聚合模型
- 现有 `Room -> ApplicationSummary` 映射层

### 4.5 首版先不做

- 复杂的跨应用仪表盘
- 多维统计图
- 高级标签系统
- 多项目聚合管理

## 5. Application Overview

### 5.1 要解决的问题

用户进入一个应用后，当前系统仍然缺少一眼看懂的“总览页”。

现在的问题是：

- 项目、Agent、Workflow、运行态是分散的
- 用户不知道当前应用是否已经可运行
- 用户不知道下一步最应该做什么

### 5.2 对用户的价值

`Application Overview` 是整个新产品最关键的解释页。

它的价值在于：

- 告诉用户这个应用当前是什么状态
- 告诉用户离运行还差什么
- 告诉用户应该去哪个分区继续

### 5.3 首版范围

首版建议包含：

- 应用头部信息
- 当前状态与状态原因
- 主项目摘要
- Agent 配置摘要
- Workflow 摘要
- 最近一次运行摘要
- 下一步推荐动作

### 5.4 优先复用的现有基础

- 现有 `ApplicationDetailView.vue`
- `ApplicationOverviewPanel.vue`
- `ApplicationHeader.vue`
- `ApplicationSectionNav.vue`
- 现有应用详情聚合模型

### 5.5 首版先不做

- 复杂的应用健康评分
- 长链路分析图
- 高级趋势预测

## 6. Projects Workspace

### 6.1 要解决的问题

当前项目能力已经存在，但它还附着在群聊工作台内部，用户难以把它当成应用正式组成部分。

当前缺口包括：

- 主项目不够正式
- 文件上下文入口不稳定
- Git 信息没有成为应用内正式视图
- 非代码目录场景缺少表达

### 6.2 对用户的价值

`Projects Workspace` 应让用户清楚知道：

- 这个应用现在绑定了什么真实资产
- 项目目录是否正确
- 是否有 Git 能力
- 产物将回写到哪里

### 6.3 首版范围

首版建议包含：

- 主项目卡片
- 项目来源与路径摘要
- Git 状态摘要
- 文件树入口
- 权限范围摘要
- 重新绑定或补绑项目的入口

### 6.4 优先复用的现有基础

- 服务端 `project` service
- `bindLocalProject / bindRoomProject`
- `getProjectDetail / listProjectFiles / getProjectGitStatus`
- 当前 `group-chat store` 中已有的项目读取逻辑

### 6.5 首版先不做

- 多项目复杂编组
- 远程 Git 全量接入
- 在线编辑器重构
- 高风险自动提交 / 自动推送策略中心

## 7. Agents Workspace

### 7.1 要解决的问题

当前用户能在群聊里配 Agent，但还不容易从产品上理解：

- 每个 Agent 负责什么
- 每个 Agent 用什么策略
- 哪些 Agent 是当前应用必须的
- 角色之间如何分工

### 7.2 对用户的价值

`Agents Workspace` 的核心价值不是“选模型”，而是“配置团队分工”。

用户应能在这里理解：

- 谁负责规划
- 谁负责执行
- 谁负责审核
- 谁负责产物整理

### 7.3 首版范围

首版建议包含：

- Agent 列表
- Agent 角色摘要
- 技能 / 模型策略摘要
- 启用状态
- 与工作流节点的关联说明
- 新增 / 编辑 / 删除 Agent 的正式入口

### 7.4 优先复用的现有基础

- `listAgents / addAgent / updateAgent / removeAgent`
- 现有 `CreateRoomForm` 中角色配置能力
- 现有群聊 Agent 配置与默认 Agents 能力

### 7.5 首版先不做

- 复杂的企业级权限矩阵
- Agent 版本库
- 跨应用角色市场

## 8. Workflow Workspace

### 8.1 要解决的问题

当前工作流能力已经存在，但仍然偏向房间内部的 beta 体验。

主要问题：

- 工作流还是“群聊里的一项功能”
- 节点、审批、角色分工没有成为正式业务配置主线
- 不同场景缺少流程模板表达

### 8.2 对用户的价值

`Workflow Workspace` 应让用户清楚看到：

- 这项工作如何推进
- 谁在什么阶段接力
- 哪些节点会产生产物
- 哪些节点需要审批或确认

### 8.3 首版范围

首版建议包含：

- 当前工作流模板摘要
- 节点列表或简化图
- 角色分配摘要
- 审批点摘要
- 切换或编辑工作流入口
- 从模板创建的入口

### 8.4 优先复用的现有基础

- `listWorkflowTemplates / getWorkflowTemplate`
- `saveWorkflowTemplate`
- 当前 workflow runtime 与节点运行态
- 现有房间工作流配置能力

### 8.5 首版先不做

- 完整新流程编辑器重写
- 高级自动编排推荐
- 复杂 BPM 级能力

## 9. Artifacts Workspace

### 9.1 要解决的问题

当前产物已经存在，但仍然太像“协作过程中的附属结果”。

这会导致：

- 用户难以专门查看一个应用的输出物
- 文档、PPT、报告等非代码结果难以成为主视图
- 运行和产物之间关系不够直观

### 9.2 对用户的价值

`Artifacts Workspace` 应把“产出什么”变成应用主线的一部分。

用户需要在这里明确看到：

- 已经产出了什么
- 哪些产物最新
- 哪些产物待审核
- 产物来自哪次运行

### 9.3 首版范围

首版建议包含：

- 产物列表
- 按类型过滤
- 最近更新
- 待确认 / 待审核标识
- 与运行记录的关联
- 基础预览或跳转入口

### 9.4 优先复用的现有基础

- `listRoomArtifacts`
- `getRoomArtifactContent`
- 当前 artifact root 目录信息
- 群聊内已有产物读取逻辑

### 9.5 首版先不做

- 完整版本对比系统
- 云端分享中心
- 富文本在线协同编辑

## 10. Runs Workspace

### 10.1 要解决的问题

当前运行态存在，但还没有成为正式的应用过程视图。

这会导致：

- 用户很难横向看“这个应用最近跑过什么”
- 失败、等待审批、已完成状态不够集中
- 运行日志和结果关系不够清楚

### 10.2 对用户的价值

`Runs Workspace` 应承接“过程视角”。

用户在这里要能回答：

- 最近执行了哪些任务
- 哪次还在跑
- 哪次失败了
- 哪次产出了关键结果

### 10.3 首版范围

首版建议包含：

- 最近运行列表
- 当前状态标签
- 开始时间 / 最近更新时间
- 失败与待审批筛选
- 进入对应协作视图或产物视图的入口

### 10.4 优先复用的现有基础

- 当前 workflow run state
- node run state
- 房间 runtime summary
- 应用详情聚合层中的 run 摘要

### 10.5 首版先不做

- 全量运行分析面板
- 高级 SLA 监控
- 复杂重跑编排策略

## 11. Collaboration Workspace

### 11.1 要解决的问题

当前协作能力最强，但它还承担了太多不该由它单独承担的顶层职责。

因此这里要解决的不是“有没有协作”，而是“把协作放回正确位置”。

### 11.2 对用户的价值

`Collaboration Workspace` 仍然是应用推进的实时现场。

用户在这里完成：

- 多角色对话
- 即时推动流程
- 查看实时消息与上下文状态
- 处理执行中的问题

### 11.3 首版范围

首版建议保持：

- 直接复用现有 `GroupChatPanel`
- 在应用详情中作为正式分区承接
- 做好从 `Overview / Runs / Artifacts` 跳入协作视图的链路

### 11.4 优先复用的现有基础

- `GroupChatPanel`
- `useGroupChatStore`
- 现有 socket runtime
- 现有 room join / runtime / artifact / workflow 逻辑

### 11.5 首版先不做

- 重写协作消息内核
- 替换 socket 架构
- 单独拆新的实时协作产品线

## 12. 模块优先级建议

建议后续模块优先级如下：

### P1

- `Applications Home`
- `Application Overview`
- `Projects Workspace`
- `Agents Workspace`

### P2

- `Workflow Workspace`
- `Artifacts Workspace`
- `Runs Workspace`

### P3

- `Collaboration Workspace` 的深度重构

说明：

- `Collaboration` 现在已经能用，所以短期不该重写
- 反而最需要补的是那些尚未成为正式应用分区的能力

## 13. 第一批功能真正要解决的用户痛点

如果从用户感知来总结，下一阶段最应该解决的是四个痛点：

- 我不知道该从哪个应用继续开始
- 我不知道当前应用还差什么才能跑
- 我不知道项目、角色、流程和结果之间是什么关系
- 我做文档、PPT、研究时，系统看起来还像开发工具

后续所有功能设计都应该回扣这四个痛点。

## 14. 本轮结论

### 14.1 下一阶段不是再扩系统页

而是要把应用工作主线补完整。

### 14.2 最关键的功能不是协作重写

而是把：

- `Projects`
- `Agents`
- `Workflow`
- `Artifacts`
- `Runs`

从旧群聊内核中正式提出来。

### 14.3 首版必须强调“清楚”和“可持续工作”

而不是追求一次性把每个模块都做得很深。
