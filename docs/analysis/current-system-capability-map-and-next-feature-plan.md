# 基于现有系统的能力地图与下一阶段功能规划

## 1. 本轮目的

这一轮不是继续抽象新概念，而是把当前仓库已经具备的系统能力、它们与新产品目标之间的关系、以及后续真正应该做的功能规划落到一张基于现状的地图里。

目标只有三个：

- 说清楚当前系统已经有什么
- 说清楚当前系统距离“应用中心化通用执行平台”还差什么
- 说清楚下一阶段最值得做的功能应该是什么

## 2. 当前系统的真实结构

结合现有前后端代码，当前系统本质上不是一个从零开始的平台，而是一个已经具备多项执行能力的 Hermes 控制台。

### 2.1 前端当前的两层结构已经初步形成

当前前端路由已经出现两层结构：

- 旧域：`/hermes/*`
- 新域：`/workbench/*`

其中：

- `/hermes/*` 仍然承载聊天、群聊、文件、模型、网关、渠道、设置等既有控制台能力
- `/workbench/*` 已开始承载应用中心化工作台壳层

这说明当前仓库已经不再是纯旧结构，第一阶段迁移已经有明确落点。

### 2.2 旧系统能力并不是空的，而是已经很厚

当前 `hermes` 域已经具备以下成熟能力：

- 单会话聊天
- 群聊协作
- 项目绑定与项目文件浏览
- 项目 Git 状态读取
- Agent 管理
- Workflow 模板与运行态
- Artifact 浏览
- Jobs / Kanban / Skills / Memory / Terminal
- Profiles / Gateways / Models / Channels / Settings

这意味着后续规划不能假装这是一个“只有聊天页的老项目”，而应该把它当作一个“能力很散、但能力本身并不弱”的系统来重组。

### 2.3 `group-chat` 其实是当前最接近未来应用容器的内核

当前最关键的既有能力不是普通 chat，而是 `group-chat`：

- 有房间语义
- 有 Agent 列表
- 有项目绑定
- 有工作流
- 有运行态
- 有 Artifact
- 有协作消息流

也就是说，当前系统最接近未来 `Application` 的，不是任意一个设置页，而是 `Room + GroupChat runtime` 这一组能力。

这也是为什么第一阶段可以采用：

- 新工作台壳
- 旧群聊协作内核

而不需要一开始就推翻重写。

### 2.4 后端已经具备多个可复用的基础服务

当前服务端并不只是一个转发层，而是已经具备多块可复用基础设施：

- `group-chat` 服务：协作、Agent、workflow、artifact
- `project` 服务：本地项目绑定、文件读取、Git 状态、权限配置
- `gateway-manager`：网关运行管理
- `profiles` / `providers` / `models`：模型基础设施
- `sessions` / `conversations`：聊天历史与同步

所以未来新产品的主要工作重点不是“从零补能力”，而是“把现有能力重新编排成新的顶层业务模型”。

## 3. 当前系统已经能支撑什么产品方向

基于现有能力，当前系统已经足以支撑以下产品方向的第一阶段：

### 3.1 可以支撑“应用中心化”

因为当前已有：

- 房间作为协作容器
- 项目作为资产上下文
- Agent 作为执行角色
- Workflow 作为协作编排
- Artifact 作为输出结果

所以把 `Application` 提升为顶层业务对象是顺势而为，而不是凭空发明。

### 3.2 可以支撑“任意项目场景”

因为当前 `project` 能力本质上绑定的是目录，而不是只绑定某一种代码后端。

这意味着从模型上，当前系统天然可以承接：

- 代码仓库
- 文档目录
- PPT 材料目录
- 研究资料目录

真正缺的不是底层抽象，而是上层产品表达和针对不同场景的模板化体验。

### 3.3 可以支撑“多角色执行”

因为当前群聊体系已经具备：

- 多 Agent 同房间
- mention 路由
- workflow 运行
- 节点状态
- 审批/执行态

所以未来的“机器人协作执行平台”并不需要另起一套完全新的运行内核。

## 4. 当前系统距离目标还差什么

虽然能力已经很多，但距离“应用中心化通用项目执行平台”仍然有几类核心缺口。

### 4.1 缺顶层业务入口

当前旧系统的主入口仍然是功能入口，不是业务入口。

表现为：

- 用户先看到聊天、模型、网关、设置
- 用户不会自然理解“如何围绕一个真实项目持续工作”
- 业务上下文被拆散在多个页面

所以当前第一缺口不是能力，而是入口。

### 4.2 缺统一业务对象表达

虽然房间、项目、Agent、Workflow、Artifact 都已经存在，但用户看到的仍然是分散对象，而不是完整的“一个应用”。

这会造成三个问题：

- 用户理解成本高
- 非代码场景迁移困难
- 后续品牌升级没有统一承载对象

### 4.3 缺场景模板化能力

当前系统已经有底层执行能力，但缺针对不同场景的最小模板化体验。

例如：

- 代码开发应用应该默认有哪些角色和流程
- 文档写作应用应该默认有哪些输出物
- PPT 应用应该默认有哪些阶段

没有这些模板，系统虽然通用，但用户仍然需要自己拼底层零件。

### 4.4 缺正式的 Run 与 Artifact 中心表达

当前运行态和产物都偏嵌在群聊工作流语义里。

这会导致：

- 运行历史不够清晰
- 产物难以成为独立主线
- 非聊天型任务很难建立清晰的结果视图

所以后续必须把：

- Run
- Artifact

从“协作细节”升级为“应用主视图能力”。

### 4.5 缺针对非代码场景的应用层引导

当前底层 `project` 抽象已经可以绑定目录，但 UI 和引导还明显偏代码协作。

这会让：

- 做文档的人感觉系统像开发工具
- 做 PPT 的人不知道从哪开始
- 做研究的人不知道输出物如何组织

因此未来的通用性，不是靠修改项目底层抽象，而是靠应用层模板、导航和产物视图来实现。

## 5. 基于现状重新定义边界

结合当前系统，后续边界建议如下。

### 5.1 Application

`Application` 是用户工作主入口，也是未来品牌的核心承载对象。

它负责组织：

- 目标
- 场景
- 主项目
- Agent
- Workflow
- Runs
- Artifacts
- Collaboration

### 5.2 Project

`Project` 是真实资产源，不等于代码仓库。

它负责提供：

- 文件上下文
- 目录结构
- Git 信息（如果有）
- 权限控制
- 产物落盘位置

### 5.3 Agent

`Agent` 是执行角色，不是模型实例。

它由以下几部分组成：

- 角色定位
- 技能集
- 模型策略
- 权限边界

### 5.4 Workflow

`Workflow` 是执行编排层，不是底层网关能力。

它负责定义：

- 谁先做
- 谁审批
- 谁继续
- 哪一步产生什么产物

### 5.5 Run

`Run` 是一次应用级任务执行。

后续它应该成为独立主线，而不是只作为群聊中的附属状态。

### 5.6 Artifact

`Artifact` 是统一输出物抽象。

它应该统一承载：

- 代码修改
- 设计草案
- PRD 文档
- 报告
- PPT 大纲
- 审核意见

### 5.7 Gateway / Provider / Model

这三者属于系统基础设施层，而不是业务工作台主入口。

后续它们应继续存在，但在产品表达上退到：

- `System`
- `Resources`
- Agent 配置内部

而不应继续占据用户默认第一入口。

## 6. 下一阶段真正应该做哪些功能

基于当前代码和既有实现，下一阶段最值得做的功能可以分成三组。

### 6.1 第一优先级：把“应用工作台”真正做成立

这是当前最关键的一组，因为它直接决定产品方向是否成立。

应继续完善：

- 应用列表页
- 应用详情页
- 创建应用向导
- 应用状态与下一步引导
- 应用内分区导航

其中重点不是视觉堆叠，而是让用户第一次能“围绕应用持续工作”。

### 6.2 第二优先级：把既有群聊能力拆成正式业务分区

这一步是从“旧能力复用”过渡到“新结构稳定”的关键。

应优先正式化的分区包括：

- Projects
- Agents
- Workflow
- Artifacts
- Runs
- Collaboration

优先顺序建议：

1. `Overview`
2. `Projects`
3. `Agents`
4. `Workflow`
5. `Artifacts`
6. `Runs`
7. `Collaboration`

说明：

- `Collaboration` 现在已经很强，短期可复用
- 真正薄弱的是 `Projects / Agents / Workflow / Artifacts / Runs` 这些业务入口页还不够正式

### 6.3 第三优先级：让系统真正适配不同场景

这一步决定它是不是通用执行平台，而不是只会服务代码开发。

建议后续补的能力包括：

- 应用场景模板
- 场景化创建向导
- 场景化 Agent 预设
- 场景化 Workflow 预设
- 场景化 Artifact 结构

第一批建议先覆盖：

- 代码开发
- 文档写作
- PPT 产出
- 研究分析

## 7. 推荐的下一阶段功能设计重点

如果从研发投入产出比来看，后续最值得重点设计的不是再补系统设置，而是以下几项。

### 7.1 Applications Home

目标：

- 成为默认首页
- 显示最近应用、推荐继续、最近运行、最近产物
- 让用户一进来就知道接下来做什么

### 7.2 Application Overview

目标：

- 展示当前目标
- 展示准备状态
- 展示主项目、角色、流程是否就绪
- 展示“下一步推荐动作”

这是把复杂系统解释清楚的关键页。

### 7.3 Projects Workspace

目标：

- 把当前项目绑定、文件、Git 信息从“协作侧能力”提升为“应用主分区”
- 支持未来扩展多项目和非 Git 项目

### 7.4 Agents Workspace

目标：

- 把角色配置做成正式分区
- 让用户先理解“角色怎么分工”，再理解模型怎么选

### 7.5 Workflow Workspace

目标：

- 把模板、节点、审批和执行图统一收敛
- 从“beta 群聊工作流”升级为“应用执行编排中心”

### 7.6 Runs & Artifacts

目标：

- 让执行记录和结果物可以脱离聊天流被查看、追踪和复盘
- 让非代码工作成果也能自然展示

## 8. 对品牌升级和 UI 重做的现实约束

当前系统已经开始有 `workbench` 路由和新导航，但整体视觉和信息架构仍然混合着旧控制台语言。

因此品牌与界面升级应遵循：

- 先用应用中心化结构统一产品主线
- 再以 `Applications / Application Detail` 作为新品牌主页面
- 再逐步弱化旧的 `Hermes control panel` 呈现方式

不要反过来只做视觉翻皮。

## 9. 当前阶段最重要的判断

结合现有系统，当前阶段最重要的判断可以浓缩成一句话：

当前项目最缺的不是底层能力，而是把现有能力收束成“一个围绕应用持续执行项目”的产品形态。

因此后续研发重点应该放在：

- 业务入口重组
- 领域对象升维
- 分区正式化
- 场景模板化

而不是优先去追求大规模底层重构。

## 10. 本轮结论

### 10.1 当前系统最值得保留的内核

- `Room + GroupChat runtime`
- `Project service`
- `Workflow runtime`
- `Gateway / Profile / Model` 基础设施

### 10.2 当前系统最需要重做的层

- 顶层业务入口
- 应用化对象表达
- 非代码场景引导
- Run / Artifact 正式主线

### 10.3 接下来最值得继续推进的事情

- 继续完善应用工作台首页与详情页
- 正式拆出 `Projects / Agents / Workflow / Artifacts / Runs`
- 为代码、文档、PPT、研究提供场景模板
- 在此基础上再做品牌和 UI 系统升级

## 11. 2026-05-30 现状功能与流程复盘

### 11.1 当前已经形成的主流程

当前产品已经形成一条比较清晰的应用主链路：

- 登录
- 进入 `Applications`
- 创建应用
- 打开应用详情
- 在 `Projects / Agents / Workflow / Collaboration / Artifacts / Runs / Settings` 之间推进

这说明产品顶层入口已经从“工具面板集合”开始转向“围绕应用持续执行”的形态。

### 11.2 当前真实的数据与能力来源

从代码结构看，当前 `workbench` 不是一个完全独立的新后端，而是建立在旧能力之上的聚合层：

- 应用对象本质上仍然聚合自 `group-chat room`
- 项目能力来自 `project service`
- 执行流能力来自 `group-chat workflow`
- 运行状态、审批和产物来自 `group-chat runtime`
- 共享技能、记忆、profile、provider、gateway 仍然来自旧 Hermes 控制面

也就是说，当前新工作台更像“应用化产品编排层”，还不是“领域彻底重构后的独立应用平台”。

### 11.3 当前已闭环的功能

目前可以认为已经有可操作闭环的部分包括：

- 应用列表与筛选
- 应用创建
- 本地项目/已有项目绑定
- 角色配置
- 工作流模板选择与工作流保存
- 协作房间接入
- 产物浏览
- 运行审批、驳回、取消
- 应用设置中的邀请码、压缩规则、流程护栏、项目权限、角色基线

这些能力已经不是纯展示壳，而是能实际写入状态。

### 11.4 当前最明显的流程断点

当前最需要注意的，不是“有没有页面”，而是“流程是否像正式应用产品一样完整”。

最关键的断点有 4 个：

- `Ready` 之后没有独立的“开始执行”动作  
  当前应用头部的主按钮在 `ready` 状态下文案是“开始执行”，但实际只是导航到 `collaboration` 分区；真正执行仍然依赖用户在群聊里发送消息或 @ 机器人触发。
- 应用对象还不是一等实体  
  当前应用仍然以 room 聚合为核心，因此应用、协作房间、运行流三者在用户心智上还没有彻底解耦。
- `Resources / System` 仍然是跨应用共享底座，而不是应用内闭环能力  
  它们现在更像“桥接页”和“说明页”，帮助用户理解继承来源，但不是真正的应用私域资源中心。
- 旧 Hermes 页仍然是部分能力的源头  
  例如 profile、models、skills、memory、channels 等，工作台已经能引导过去，但还没有彻底吸收到“应用产品主线”里。

### 11.5 当前最真实的产品流程判断

如果用一句话总结当前阶段：

现在这个产品已经完成了“以应用为中心重新组织能力”的第一阶段，
但离“任何项目、任何场景都能像正式执行系统一样自然推进”的目标，还差最后一层运行流程正式化。

换句话说：

- 结构已经像应用平台
- 执行方式还带着群聊工作台基因

### 11.6 接下来最值得优先打磨的流程

建议优先打磨以下顺序：

1. `Ready -> Start Run -> Running -> Review -> Completed` 的正式执行动作闭环
2. 应用详情页顶部主动作和各分区动作的职责收敛
3. 把协作区从“底层实现入口”提升为“执行现场”，而不是唯一入口
4. 把共享资源的继承关系进一步产品化，减少用户在新旧页面之间跳转

### 11.7 对后续研发的启发

这次复盘说明，后续项目打磨不应该再平均发力，而应该集中在：

- 运行动作正式化
- 应用实体升格
- 新旧能力边界收口
- 跨场景模板化

只有这四件事继续推进，代码、文档、PPT、研究这些场景才会真正共用同一套应用平台心智。

## 11.8 本轮落地：正式启动执行链路已补齐

围绕上一轮识别出的最大断点，本轮已经把 `Ready -> Start Run` 从“伪按钮”补成正式产品动作：

- 后端新增 `POST /api/hermes/group-chat/rooms/:roomId/start-run`
- 启动动作会统一处理：
  - 房间 / 工作流存在性校验
  - 工作流角色与房间 AI 对齐校验
  - 待审批节点冲突校验
  - 房间自动激活
  - 写入 kickoff 消息
  - 复用现有 workflow 引擎推进到首个执行节点
- 前端应用详情页头部主按钮已从“仅导航到 collaboration”改为“真实启动执行”
- 启动成功后会刷新应用详情状态，并自动进入协作现场

这意味着：

- `Start Execution` 文案现在和真实行为一致
- 协作区从“唯一触发入口”下降为“执行现场”
- 应用视角与群聊运行内核之间第一次建立了正式桥梁

## 11.9 当前下一优先级

在正式启动动作补齐后，接下来最值得继续打磨的是：

1. Run 输入建模  
   当前启动使用默认 kickoff 指令，下一步可以把“本轮执行目标 / 补充说明 / 输入材料”显式产品化。
2. Run 恢复与复跑  
   已支持启动与取消，但还缺应用级“重新执行 / 基于当前产物继续执行”的清晰入口。
3. 应用与房间边界进一步澄清  
   当前虽然已有正式 run start，但应用实体仍然来自 room 聚合，后续还可以继续抽离。

## 11.10 本轮落地：Runs 工作区升级为运行控制台

在补齐正式启动动作后，本轮继续把 `Runs` 工作区从“审批面板”提升为“运行控制台”：

- 当应用尚未开始、上一轮已完成、或上一轮失败时
- 用户现在都可以直接在 `Runs` 工作区里发起下一轮执行
- 发起前可填写“本轮运行说明 / Run Brief”

这一步解决的是另一个真实使用痛点：

- 以前运行页只能看状态、审批、取消
- 但不能承担“重新发起 / 带说明发起下一轮”的职责
- 这会导致运行页像监控页，而不像控制台

现在的产品语义更合理了：

- Overview 负责理解应用状态
- Collaboration 负责执行现场
- Runs 负责执行生命周期控制

## 11.11 当前产品主线更完整后的判断

## 11.26 本轮补充：本地运行环境与数据视角隔离

本轮又暴露出一个很实际但很重要的问题：

- 当前仓库代码可以启动成功
- 但运行时读取的持久化数据目录，可能并不是当前这轮研发真正想验证的那一套数据

### 11.26.1 暴露出来的风险

如果继续默认把本地验证直接连到全局 `~/.hermes-web-ui`，就会出现一种很危险的假象：

- 代码是当前仓库的
- 但应用列表、run 历史、artifact 语义，实际上来自另一套历史运行环境

### 11.26.2 本轮形成的新工程原则

从这一轮开始，需要明确区分两件事：

- 代码视角：当前仓库里正在开发的实现
- 数据视角：当前实例实际连接到的持久化状态

本地研发时，如果目标是验证当前仓库功能，应优先使用仓库内隔离数据目录，而不是默认复用全局用户目录。

### 11.26.3 本轮新增能力

本轮已补上：

- `HERMES_WEBUI_HOME` / `WEBUI_HOME` 显式指定 Web UI 数据目录
- `npm run dev:isolated`
- 文档中的仓库隔离运行说明

### 11.26.4 研发价值

这不是单纯的启动脚本小修，而是在补“研发验证边界”：

- 避免把历史数据误当成当前应用状态
- 避免不同 daemon / checkout 之间互相串台
- 让后续 workbench、run、artifact 的打磨建立在可控的数据面上

## 11.27 本轮补充：启动链路可用性与预置团队真实落库

本轮在真实隔离实例验证时，又暴露出两个会直接影响产品判断的问题：

- Web UI 会在监听前等待网关探测与自动启动，导致页面本身可能起不来
- 预置应用虽然有默认角色基线，但活动团队表里可能仍是空的

### 11.27.1 启动链路问题的产品影响

如果 Web UI 要等网关探测与自动启动完全结束后才真正监听端口，就会出现：

- 页面打不开
- 健康检查失败
- 研发误以为 workbench 功能坏了

这会把“网关就绪问题”和“工作台页面问题”混在一起。

### 11.27.2 本轮启动链路修正

本轮已将启动链调整为：

- Web UI 先完成监听
- 网关探测与自动启动转为后台完成

这样即使网关探测失败，页面本身仍然可以打开，工作台验收也不会被阻断。

### 11.27.3 预置团队问题的产品影响

预置应用如果只保存了 `defaultAgentsJson`，但没有把角色真实写入 `gc_room_agents`，前台会出现一类误判：

- 应用看起来有流程、有默认角色
- 但概览与运行摘要会把它判成“未配置团队”或“0 成员”

这会直接削弱应用工作台对“预置可用应用”的可信表达。

### 11.27.4 本轮预置团队修正

本轮已补上：

- 预置房间 seed 时，同步把角色写入 `gc_room_agents`

真实隔离实例验证结果：

- `preset-opc-rd-team` 已有 7 个真实团队成员
- room runtime 摘要中的 `totalAgents` 已变为 7

## 11.28 本轮补充：区分团队配置完成与机器人在线状态

在 11.27 补齐预置团队数据后，又暴露出一个状态语义问题：

- 房间处于离线待机
- 但运行摘要仍可能显示机器人在线

### 11.28.1 问题本质

团队配置完成和机器人在线，是两个不同概念：

- `totalAgents` 表示这个应用团队配置了多少角色
- `onlineAgents` 表示当前激活后真正上线的机器人数量

如果离线待机状态下仍显示 `onlineAgents > 0`，用户会误以为机器人已经在工作，实际它们不会响应。

### 11.28.2 本轮修正

本轮已调整：

- 预置房间同步时，离线房间只补齐团队数据，不自动上线机器人
- runtime 摘要以房间激活状态为准，未激活时 `onlineAgents = 0`

### 11.28.3 验证结果

真实隔离实例验证：

- `isActive = false`
- `totalAgents = 7`
- `onlineAgents = 0`

### 11.28.4 产品意义

这一步让工作台能准确表达：

- 应用团队是否已经配置完成
- 当前机器人是否真的在线可响应

二者拆开后，`Runs`、`Collaboration` 和运行状态提示才不会给用户错误预期。

### 11.27.5 产品价值

这一轮补的是两层基础可用性：

- 启动链路可用性：让页面与网关生命周期解耦
- 预置应用可信性：让预置应用不再只是“看起来有团队”，而是真正有团队数据

### 11.29.2 外部用户最容易卡住的地方

外部用户第一次使用时，可能会遇到以下理解断点：

- 不知道“应用”到底对应真实世界里的什么对象
- 不知道代码、文档、PPT、研究场景应该如何开始

- 不知道应用设置、共享资源、系统控制三层边界分别解决什么问题

### 11.29.3 下一阶段产品解释策略

建议统一采用以下解释路径：

> 一个应用就是一个真实交付目标的 AI 工作区。

它包含：

- 资产：本地项目、远程项目、资料目录或空白上下文
- 团队：围绕这个交付目标配置的机器人角色
- 流程：从需求、分析、设计、执行到验收的阶段规则
- 运行：每次正式推进工作的一轮 Run
- 产物：每次 Run 产生、可审阅、可继续加工的文件或结果

这个解释比单独讲网关、模型、机器人、项目更适合外部用户。

### 11.29.4 精调优先级

P0：理解闭环

- 应用总览补强就绪诊断
- Runs 首次运行空态解释
- 协作待机/在线状态解释
- 创建向导说明“场景 = 交付目标模板”

P1：交付闭环

- 场景默认角色模板
- 场景默认执行流模板
- 场景默认产物目录
- Run 与 Artifact 的来源关系可视化

P2：复用闭环

- 模板市场
- 跨应用资源继承可视化
- 组织级角色/流程/技能基线
- 发布、分享、权限与团队协作能力

### 11.29.5 对当前研发节奏的影响

这轮分析建议下一步不要立刻做大规模架构迁移，而是优先做小而关键的产品精调：

- 总览页更清楚

这几个点完成后，外部用户才能自然理解系统价值：它不是一个聊天工具，也不是一个后台控制台，而是一个能把任意项目场景持续推进到交付结果的 AI 工作台。
