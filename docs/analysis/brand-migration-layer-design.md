# 品牌迁移分层设计

## 1. 本轮目的

在已经完成应用工作台主链路和视觉基线收敛后，需要正式把“未来不再沿用 `Hermes Web UI` 名称”拆成可执行的迁移层，而不是直接大范围重命名仓库。

本轮目标是先明确：

- 哪些属于前台可见品牌层
- 哪些属于运行时技术契约
- 哪些可以立即调整
- 哪些必须留到后续专项迁移

## 2. 当前判断

当前系统实际上已经出现了两层并存：

### 2.1 前台品牌层已经部分可切换

客户端已经存在：

- `packages/client/src/constants/branding.ts`

并且 `AppSidebar`、`App.vue`、消息空态等位置已经开始引用 `BRAND_NAME / BRAND_FULL_NAME / BRAND_LOGO_PATH`。

这说明 UI 可见品牌文案和 logo，已经具备独立迁移条件。

### 2.2 运行时层仍深度绑定 Hermes

以下内容目前仍属于技术与运行时契约，不应在这一阶段直接修改：

- CLI 包名 `hermes-web-ui`
- `bin/hermes-web-ui.mjs`
- `~/.hermes-web-ui`
- `hermes/` 路由前缀
- server 侧 `hermes/*` 模块命名
- Hermes gateway / Hermes profile / Hermes CLI 数据结构

这些部分不仅是文案，还影响：

- 本地启动方式
- 数据目录兼容
- 自动更新
- 文档安装指令
- server/client API 协议

## 3. 本轮设计结论

### 3.1 品牌迁移必须分层推进

建议把迁移拆成三层：

1. **展示层迁移**
   - UI 文案
   - logo
   - 页面标题
   - 导航命名
   - 空状态与产品说明

2. **产品语言迁移**
   - 从“控制台/后台”话术转为“工作台/执行平台”话术
   - 优先体现在导航、hero、概览说明和空状态引导中

3. **运行时契约迁移**
   - 包名
   - CLI
   - 数据目录
   - 官网与 README 安装说明
   - server / route / openapi 命名

当前阶段只建议继续推进前两层，不建议直接进入第三层。

### 3.2 运行时 Hermes 与产品品牌可以暂时并存

中期更合理的状态不是“立即去 Hermes 化”，而是：

- 对用户可见产品，逐步突出新的 workbench 品牌
- 对底层执行内核，继续保留 Hermes 运行时身份

可以把它理解成：

- 前台：新的通用应用工作台
- 后台：仍然复用 Hermes runtime / gateway / profile / CLI

这也符合当前“新壳 + 旧协作核心”的总体迁移策略。

## 4. 当前最适合立即做的事情

基于上面的判断，当前最适合先推进的是：

- workbench 顶层导航文案收口
- 页面标题与说明文案收口
- 统一品牌常量在前台 UI 的使用范围
- 补齐品牌迁移说明文档

## 5. 下一阶段建议

在这层设计明确后，下一阶段可以继续：

1. 调整工作台导航与外壳文案
2. 扩展 `branding.ts` 为更完整的品牌配置层
3. 单独设计“CLI / 包名 / 数据目录兼容迁移方案”

这样既不会破坏当前系统的可运行性，也能持续把产品从旧品牌壳中抽离出来。
